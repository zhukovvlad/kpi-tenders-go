# 2026-04-30 — Рефакторинг семантической экстракции вокруг `extraction_requests`

## Контекст

Эндпойнт `POST /api/v1/documents/:id/extract` сразу запускал `resolve_keys`, минуя `convert` и `anonymize`. Тогда `triggerExtract` падал на проверке `payload.MDDocumentID == ""` (Python никогда не возвращал этот ключ — только `{new_keys, resolved_schema}`), цепочка тихо обрывалась, задача висела «в выполнении» с точки зрения клиента. Документация (`python-kpi-tenders/CLAUDE.md`) прямо говорила «Go сам знает, какой документ отдать на extract», но Go-код требовал id из payload — конфликт документации и реализации.

Дополнительно требовалось:
- сделать анонимизацию опциональной (галочка в UI, default включена);
- допустить параллельные запросы на один документ (с разными вопросами и разными флагами анонимизации) с независимыми ответами.

## Что сделано

### Миграция `000003_extraction_requests`

- Новая таблица `extraction_requests(id, document_id, organization_id, questions jsonb, anonymize bool, status, resolved_schema jsonb, error_message, ...)`.
- Composite FK `(document_id, organization_id) → documents(id, organization_id)` для tenant isolation; CHECK на `questions` как непустой jsonb-массив; immut org_id триггер.
- В `document_tasks` добавлена колонка `extraction_request_id` (FK CASCADE).
- Старый `UNIQUE(document_id, module_name)` снесён, заменён двумя partial-индексами:
  - `uq_document_tasks_doc_singleton (document_id, module_name) WHERE module_name IN ('convert','anonymize')` — singleton артефактов на документ.
  - `uq_document_tasks_request_module (extraction_request_id, module_name) WHERE module_name IN ('resolve_keys','extract')` — singleton на запрос.
- CHECK `document_tasks_module_request_chk` форсит инвариант: `convert/anonymize ⇔ extraction_request_id IS NULL`, `resolve_keys/extract ⇔ NOT NULL`.

### sqlc-запросы

- Новый файл `sql/query/extraction_request.sql`: `CreateExtractionRequest`, `GetExtractionRequest` (tenant-scoped) / `GetExtractionRequestByID` (internal — без org-check для callback'ов), `ListPendingExtractionRequestsByDocument`, `SetExtractionRequestStatus`, `SetExtractionRequestResolvedSchema`.
- В `sql/query/document_task.sql` `CreateDocumentTaskInternal` распался на `CreateDocumentTaskSingleton` (convert/anonymize) и `CreateDocumentTaskForRequest` (resolve_keys/extract). Каждый бьёт по своему partial index — ON CONFLICT с явным `WHERE`-предикатом.
- В `sql/query/extraction.sql` добавлен `ListExtractedDataForKeys` — JOIN ключей с экстрактом для GET-эндпойнта.

### Сервисный слой

- **`ExtractionService`** теперь владеет всем пайплайном:
  - `Initiate(ctx, docID, orgID, questions, anonymize)` валидирует документ → создаёт `extraction_request` → `Progress(req)`.
  - `Progress` смотрит существующие артефакты документа (`ListDocumentsByParent`) и решает что запустить:
    - есть нужный артефакт (`anonymize_doc` для `anonymize=true`, `convert_md` для `false`) → `enqueueResolveKeys`.
    - есть MD, но нет anonymize при `anonymize=true` → `ensureAnonymizeTask`.
    - нет MD → `ensureConvertTask`.
  - `OnResolveKeysCompleted` — upsert новых ключей, сохраняет `resolved_schema` на запросе, ищет артефакт по `parent_id+artifact_kind` и стартует extract через `CreateDocumentTaskForRequest`.
  - `OnExtractCompleted` — batch upsert `document_extracted_data`, переводит запрос в `completed`.
  - `MarkRequestFailed` — best-effort пометка `failed`.
- **`WorkerService`** получил консьюмер-сайд интерфейс `extractionPipeline` (`Progress`, `OnResolveKeysCompleted`, `OnExtractCompleted`, `MarkRequestFailed`). После `convert/anonymize completed` обходит `ListPendingExtractionRequestsByDocument` и зовёт `pipeline.Progress` для каждого запроса. resolve_keys/extract callback'и просто делегируются в pipeline. `failed`-таска с непустым `extraction_request_id` тянет за собой `MarkRequestFailed`.
- В `NewServer()` `ExtractionService` строится **до** `WorkerService` и передаётся ему как pipeline. `*pythonworker.Publisher` шарится на три сервиса.

### HTTP-слой

- `POST /api/v1/documents/:id/extract` принимает `{questions, anonymize?}` (default `true`), отвечает `201 {extraction_request_id, status}`.
- Новый `GET /api/v1/extraction-requests/:id` — tenant-scoped, возвращает запрос + `resolved_schema` + answers (через JOIN по resolved_schema → extraction_keys → document_extracted_data).
- `handler_extraction_request.go` владеет JSON-сериализацией: парсит JSONB-поля (`questions`, `resolved_schema`) и склеивает с `service.ExtractionAnswer`.

### Контракт с Python

- Из result_payload `resolve_keys` убрано требование `md_document_id` — Go находит артефакт сам через `parent_id + artifact_kind`. Контракт стал тривиальным: `{new_keys, resolved_schema}`.
- Из kwargs `extract`-задачи тоже убран `md_document_id` (Python его не читал, поле было информационным шумом).
- Никаких правок в Python-репо не требуется: `resolve_keys.py` всегда возвращал чистый payload, `extract.py` читал только `extraction_schema`.

## Решения и обоснования

- **Почему отдельная таблица `extraction_requests`, а не флаг на `document_tasks`.** Запрос экстракции — это пользовательская сущность («я задал документу 5 вопросов»), которая не равна одной таске воркера. Один запрос порождает 1–2 таски (`resolve_keys` + `extract`); параллельные запросы — независимые цепочки. Хранить вопросы и `resolved_schema` на `document_tasks` означало бы перегружать таску ролями, путать state-machine и ломать UNIQUE-семантику. Отдельная таблица даёт явный жизненный цикл (`pending → running → completed/failed`) и чистый GET-эндпойнт для UI.
- **Partial UNIQUE вместо одного на (document_id, module_name).** convert/anonymize — артефакты документа, переиспользуются между запросами. resolve_keys/extract — per-request. Это разные виды идемпотентности, и одного UNIQUE-ключа не хватает. Два partial-индекса описывают эти инварианты ровно и позволяют sqlc-генерируемому ON CONFLICT работать без хака.
- **Pipeline-интерфейс между `WorkerService` и `ExtractionService`.** Альтернатива — вкатить всю extraction-логику в WorkerService — оставила бы worker callback файл монструозным и нарушила границу «WorkerService отвечает за callbacks, ExtractionService — за пользовательский запрос». Узкий интерфейс из 4 методов добавил минимум зависимостей и легко моккается в тестах.
- **Анонимизация по convert→anonymize цепочке остаётся unconditional.** Можно было бы скипать anonymize, если ни один pending-запрос его не просит, но (а) artifact полезен сам по себе (download anonymized version), (б) потенциальная экономия LLM-вызова не перевешивает сложность ветвления. anonymize=false запросы не блокируются на anonymize: Progress переводит их в resolve_keys сразу после convert completed, anonymize крутится параллельно.

## Ловушки и особенности

- **`ON CONFLICT` с partial-индексом требует точного `WHERE`-предиката в INSERT-запросе.** В `CreateDocumentTaskSingleton`/`CreateDocumentTaskForRequest` `WHERE` повторяет тот же предикат, что и в `CREATE UNIQUE INDEX`. Иначе PG отказывается выводить индекс и фейлится с `there is no unique or exclusion constraint matching the ON CONFLICT specification`.
- **`document_tasks.input_storage_path` NOT NULL и CHECK btrim ≠ ''.** Для `resolve_keys` файл не нужен (Python не качает MinIO), но колонка всё равно должна быть заполнена. Кладём туда путь target-артефакта — это семантически осмысленный референс и проходит CHECK.
- **`pgtype.UUID{Bytes: ..., Valid: true}` против nil-интерфейсов.** При прокидывании `extraction_request_id` в `CreateDocumentTaskForRequestParams` важно собирать `pgtype.UUID` с `Valid: true`, иначе sqlc отправит NULL и попадёт под CHECK invariant'а.
- **`extraction_requests.resolved_schema` храним JSONB-снапшотом.** Можно было бы вычислять на лету через JOIN ключей по `extracted_data`, но (а) схема — артефакт LLM-вызова, и хранить её на запросе важно для аудита; (б) при множественных запросах с пересекающимися ключами это однозначно описывает «какие ключи относятся к этому запросу». GET-эндпойнт читает schema → берёт key_names → `ListExtractedDataForKeys`.
- **Best-effort на progress в `Initiate`.** Если `Progress` падает (например, Redis недоступен), запрос всё равно создан в БД и виден через GET. Следующий callback от любой связанной таски снова прокачает Progress.
