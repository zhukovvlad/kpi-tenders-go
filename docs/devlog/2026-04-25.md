# Дневник разработки — go-kpi-tenders

## 2026-04-25 — День 9: Фикс триггера Python при создании задачи

### Контекст

До этого дня цепочка обработки документа выглядела так:

```
Frontend → POST /api/v1/tasks → INSERT в document_tasks → ??? → Python
```

«???» была пропастью. `DocumentTaskService.Create` честно сохранял запись в БД, но никогда не вызывал Python `POST /process`. В итоге `convert` и `anonymize` никогда не стартовали автоматически — нужно было либо вызывать Python вручную, либо надеяться на что-то, чего не существовало.

Python-триггер был реализован только в `WorkerService` — для **chaining**: когда `convert` завершается, Go создаёт задачу `anonymize` и зовёт Python. Но **первый** вызов цепочки — `convert` — Python так и не получал.

---

### Суть проблемы

`DocumentTaskService` — сервис, который обрабатывает `POST /api/v1/tasks`. Он умел только INSERT. У него не было Python-клиента вообще.

`WorkerService` — сервис, который обрабатывает callback от Python (`PATCH /internal/worker/tasks/:id/status`). Он умел и триггерить Python (для chaining), и получать обратно результаты. Но он вступает в игру **после** того, как Python уже что-то обработал.

Цепочка никогда не стартовала, потому что первый пинок Python'у так и не происходил.

---

### Что сделано

**1. `DocumentTaskService` — добавлен Python-клиент**

Структура расширена полем `pythonClient workerPythonClient` — тот же consumer-side interface, что использует `WorkerService`. Дублировать интерфейс не нужно: он объявлен один раз в `service_worker.go` и переиспользуется внутри пакета.

Конструктор:
```go
func NewDocumentTaskService(
    repo         repository.Querier,
    pythonClient workerPythonClient, // nil — Python отключён
    log          *slog.Logger,
) *DocumentTaskService
```

**2. Метод `Create` — логика триггера после INSERT**

После успешного `CreateDocumentTask` метод:

1. Делает `GetDocument` чтобы достать `storage_path` — Python нужен путь к файлу в S3.
2. Вызывает `pythonClient.Process(...)` — тот самый `POST /process`.

Оба шага **best-effort**:
- Если `GetDocument` упал — задача уже в БД, логируем ошибку и возвращаем задачу без ошибки (пользователь может ретриггернуть).
- Если `Process` упал — то же самое: задача в БД, Python-триггер залогирован, ошибка наружу не уходит.

```
POST /api/v1/tasks
  → CreateDocumentTask (INSERT)
  → GetDocument (достать storage_path)
  → pythonClient.Process (POST /process к Python)
  → вернуть задачу клиенту
```

**3. `NewServer()` — единый экземпляр Python-клиента**

Раньше `WorkerService` создавал свой `pythonworker.New(...)` внутри `if`. Теперь клиент создаётся один раз и передаётся в оба сервиса:

```go
var pythonClient *pythonworker.Client
if cfg.PythonServiceURL != "" {
    pythonClient = pythonworker.New(cfg.PythonServiceURL)
} else {
    log.Warn("worker: PYTHON_SERVICE_URL not set — worker callback endpoint disabled")
}

srv.documentTaskService = service.NewDocumentTaskService(db, pythonClient, log)
// workerService оставляем nil если pythonClient == nil
if pythonClient != nil {
    srv.workerService = service.NewWorkerService(db, pythonClient, log)
}
```

Если `PYTHON_SERVICE_URL` не задан — оба сервиса получают `nil`. `DocumentTaskService` guard'ит через `if s.pythonClient != nil`, `WorkerService` вообще не создаётся.

### Итог

Теперь полная цепочка работает:

```
POST /api/v1/tasks
  → INSERT document_tasks
  → Python POST /process (convert)
  → Python PATCH /internal/worker/tasks/:id/status (completed)
  → INSERT document_tasks (anonymize)
  → Python POST /process (anonymize)
  → Python PATCH /internal/worker/tasks/:id/status (completed)
```

Первый и все последующие вызовы Python теперь происходят автоматически.
