# Дневник разработки — go-kpi-tenders

## 2026-04-25 — День 9: Фикс триггера Python + переход на Celery/Redis

### Часть 1: Фикс триггера Python при создании задачи

#### Контекст

До этого дня цепочка обработки документа выглядела так:

```
Frontend → POST /api/v1/tasks → INSERT в document_tasks → ??? → Python
```

«???» была пропастью. `DocumentTaskService.Create` честно сохранял запись в БД, но никогда не вызывал Python. В итоге `convert` и `anonymize` никогда не стартовали автоматически.

Python-триггер был реализован только в `WorkerService` — для **chaining**: когда `convert` завершается, Go создаёт задачу `anonymize` и зовёт Python. Но **первый** вызов цепочки — `convert` — Python так и не получал.

#### Суть проблемы

`DocumentTaskService` — сервис, который обрабатывает `POST /api/v1/tasks`. Он умел только INSERT. У него не было Python-клиента вообще.

#### Что сделано

`DocumentTaskService` расширен полем `pythonClient workerPythonClient`. После успешного INSERT метод `Create`:

1. Делает `GetDocument`, чтобы достать `storage_path`.
2. Вызывает `pythonClient.Process(...)` — best-effort: ошибки логируются, наружу не уходят.

Два новых теста:
- `TestDocumentTaskService_Create_TriggersPython_WithCorrectFields`
- `TestDocumentTaskService_Create_PythonError_ReturnsTaskWithoutError`

### Часть 2: Замена HTTP-клиента на Celery-издатель поверх Redis

#### Контекст

Python-сервис убрал HTTP-эндпоинт `POST /process`. Теперь задачи должны публиковаться напрямую в Redis как Celery-сообщения (`LPUSH`). Это повышает надёжность: если Python-воркеры недоступны, задачи всё равно попадают в очередь и будут обработаны при подъёме воркеров.

#### Что сделано

**1. `internal/pythonworker/client.go`** — полностью переписан:
- Тип `Client` (HTTP) → `Publisher` (Redis/Celery)
- `New(baseURL string) *Client` → `New(redisURL string) (*Publisher, error)` через `redis.ParseURL`
- `Process` формирует Celery protocol v2 сообщение:
  - body = JSON `[[task_id, document_id, storage_path], {}, {callbacks:null,...}]`, base64-encoded
  - headers: `lang`, `task`, `id`, `root_id`, `origin: "go-kpi-tenders"`, ...
  - properties: `correlation_id`, `delivery_mode: 2`, `body_encoding: "base64"`, ...
  - `LPUSH <queue> <json>` в Redis
- `resolveModule` маппит модуль → очередь + имя задачи:

| ModuleName | Queue | Task |
|---|---|---|
| `convert` | `io` | `app.workers.convert.convert_task` |
| `parse_invoice` | `io` | `app.workers.parse_invoice.parse_invoice_task` |
| `anonymize` | `llm` | `app.workers.anonymize.anonymize_task` |
| `extract` | `llm` | `app.workers.extract.extract_task` |

**2. `internal/server/server.go`**:
- `*pythonworker.Client` → `*pythonworker.Publisher`
- Инициализируется через `cfg.RedisURL` (обязателен; при ошибке парсинга `NewServer()` возвращает `error`, а `cmd/api/main.go` логирует и завершает процесс)
- `workerService` создаётся безусловно при успешной инициализации сервера; `srv.Close()` освобождает Redis-пул при shutdown

**3. `internal/config/config.go`**:
- Удалено поле `PythonServiceURL` и его URL-валидация
- Удалён импорт `net/url`

**4. Тест-хелперы** `newTestServer()` и `newTestServerWithJWT()` — добавлен `RedisURL: "redis://localhost:6379/0"` (только парсинг, подключения нет).

**5. Зависимость**: добавлен `github.com/redis/go-redis/v9`.

### Итог

Полная цепочка теперь работает через Redis/Celery:

```
POST /api/v1/tasks
  → INSERT document_tasks
  → LPUSH io {celery v2 message: convert}
  → Python воркер забирает задачу из очереди
  → PATCH /internal/worker/tasks/:id/status (completed)
  → INSERT document_tasks (anonymize)
  → LPUSH llm {celery v2 message: anonymize}
  → Python воркер забирает задачу
  → PATCH /internal/worker/tasks/:id/status (completed)
```

`make test-unit` — всё зелёное.
