# Дневник разработки — go-kpi-tenders

## 2026-04-14 — День 2: JWT-авторизация на HttpOnly Cookies

### Что сделано

**1. Зависимости**
- `github.com/golang-jwt/jwt/v5` добавлен как прямая зависимость
- `golang.org/x/crypto` (bcrypt) переведён из `indirect` в `direct`

**2. Конфигурация (`internal/config/config.go`)**
- Добавлены три обязательных переменных окружения:
  - `JWT_ACCESS_SECRET` — секрет для подписи access token
  - `JWT_REFRESH_SECRET` — секрет для подписи refresh token
  - `SERVICE_TOKEN` — статический токен для service-to-service аутентификации

**3. Repository: запросы к таблице users**
- `sql/query/user.sql` — два запроса: `GetUserByEmail`, `GetUserByID`
- `internal/repository/user.sql.go` — ручная реализация sqlc-стиля (scan по всем колонкам)
- `internal/repository/querier.go` — добавлены `GetUserByEmail` и `GetUserByID` в интерфейс `Querier`

**4. AuthService (`internal/service/auth.go`)**
- Структура `AuthService` с полями `repo`, `log`, `accessSecret`, `refreshSecret`
- Типы Claims:
  - `Claims` (access token) — `UserID`, `OrgID`, `Role` + `jwt.RegisteredClaims`
  - `RefreshClaims` (refresh token) — только `UserID` + `jwt.RegisteredClaims`
- Методы:
  - `Login(ctx, email, password)` — bcrypt-верификация, возвращает `ErrInvalidCredentials` в обоих случаях (нет юзера / неверный пароль) для защиты от user enumeration
  - `GenerateTokens(userID, orgID, role)` — выпускает пару access (15 мин) + refresh (30 дней), подписанных HS256
  - `ValidateAccessToken(tokenStr)` — возвращает `*Claims`; `jwt.WithExpirationRequired()` гарантирует наличие `exp`
  - `ValidateRefreshToken(tokenStr)` — возвращает `*RefreshClaims`
  - В key-функции: явная проверка `*jwt.SigningMethodHMAC` для защиты от algorithm confusion

**5. Middleware (`internal/server/middleware.go`)**
- `AuthMiddleware()` — читает `access_token` из cookie, валидирует, кладёт `userID`, `orgID`, `role` в Gin-контекст
- `ServiceBearerAuth()` — извлекает `Bearer <token>` из заголовка `Authorization`, сравнивает с `cfg.ServiceToken` через `crypto/subtle.ConstantTimeCompare` (защита от timing attack)

**6. Handlers (`internal/server/auth.go`)**
- `POST /api/v1/auth/login` — биндинг JSON, вызов `authService.Login`, запись кук
- `POST /api/v1/auth/refresh` — валидация `refresh_token` куки, `GetUserByID` для актуальных данных, выпуск новой пары
- `POST /api/v1/auth/logout` — `clearAuthCookies` (Max-Age: -1)
- Вспомогательные методы:
  - `setAuthCookies` — `access_token` на `/`, `refresh_token` scoped на `/api/v1/auth`
  - `clearAuthCookies` — сбрасывает обе куки

**7. Server (`internal/server/server.go`)**
- `authService *service.AuthService` добавлен в структуру `Server`
- `NewServer` инициализирует `authService` из конфига
- Маршруты зарегистрированы в `setupRouter`:
  ```
  POST /api/v1/auth/login
  POST /api/v1/auth/refresh
  POST /api/v1/auth/logout
  ```

### Параметры кук

| Cookie | Path | Max-Age | HttpOnly | Secure | SameSite |
|---|---|---|---|---|---|
| `access_token` | `/` | 15 мин | ✓ | prod-only | Lax |
| `refresh_token` | `/api/v1/auth` | 30 дней | ✓ | prod-only | Lax |

`Secure: true` включается когда `APP_ENV != "local"`.

### Переменные окружения (обновить `.env`)
```bash
JWT_ACCESS_SECRET=<random-32+-bytes>
JWT_REFRESH_SECRET=<random-32+-bytes>
SERVICE_TOKEN=<opaque-static-token>
```

### Принятые решения

- **HttpOnly cookie вместо localStorage** — XSS не может прочитать токен из JS
- **Refresh token scoped на `/api/v1/auth`** — браузер отправляет его только на этот путь, минимизируя поверхность атаки
- **Одинаковая ошибка при неверном email и пароле** — защита от user enumeration
- **`crypto/subtle.ConstantTimeCompare`** для `ServiceToken` — строковое `==` уязвимо к timing attack
- **`jwt.WithExpirationRequired()`** — токены без поля `exp` отклоняются
- **Явная проверка алгоритма в key-функции** — защита от algorithm confusion (`alg: none` и пр.)
- **Перечитывание юзера при refresh** — `/refresh` делает `GetUserByID`, чтобы подхватить актуальные `role`/`org` (revocation-lite)
- **Раздельные секреты** для access и refresh токенов — компрометация одного не даёт подделать другой
