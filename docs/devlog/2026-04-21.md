# Дневник разработки — go-kpi-tenders

## 2026-04-21 — День 5: Construction Sites, рефакторинг документов, tenant-изоляция

### Что сделано

**1. Новая сущность: ConstructionSite (объект строительства)**

Добавлена иерархическая сущность `construction_sites` — основная единица организации документации:

- Таблица `construction_sites`: `id`, `organization_id`, `parent_id` (самоссылка, иерархия ЖК → очереди → корпуса), `name`, `status`, `created_by`, `created_at`, `updated_at`
- SQL-запросы (`sql/query/construction_site.sql`): `CreateConstructionSite`, `GetConstructionSite`, `ListConstructionSitesByOrganization`, `UpdateConstructionSite`, `DeleteConstructionSite`
- `ConstructionSiteService` (`internal/service/service_construction_site.go`) с методами `Create`, `Get`, `ListByOrganization`, `Update`, `Delete`
- CRUD-хендлеры (`internal/server/handler_construction_site.go`) и маршруты `POST/GET/GET/:id/PATCH/:id/DELETE/:id /api/v1/sites`
- `DeleteConstructionSite` сразу реализован через `:execrows` → `not_found` при `rows == 0`

**2. Рефакторинг схемы и документов**

Миграция `000001` переработана:
- Убраны устаревшие `projects` и `catalog_positions` (перенесены в отдельную схему позже)
- `documents` — новые поля: `site_id` (FK на `construction_sites`), `parent_id` (самоссылка для версионирования), `file_name`, `storage_path`, `mime_type`, `file_size_bytes`; убраны `title`, `file_path`, `status`
- `document_tasks` — добавлены поля `module_name`, `celery_task_id`, `result_payload`, `error_message`
- Все UUID-ключи переведены с `uuid_generate_v4()` на `gen_random_uuid()` (встроен в PG13+)

**3. Реализация хендлеров документов и задач**

Заглушки `not_implemented` заменены полной реализацией:
- `CreateDocument` / `ListDocuments` (с опциональной фильтрацией по `?site_id`) / `GetDocument` / `DeleteDocument`
- `CreateDocumentTask` / `ListDocumentTasks` / `GetDocumentTask` / `UpdateDocumentTaskStatus` / `DeleteDocumentTask`

**4. Tenant-изоляция — fix по итогам code review (PR #5)**

По замечаниям CodeRabbit и Copilot устранены IDOR-уязвимости (OWASP A01):

| Уязвимость | Решение |
|---|---|
| `GetConstructionSite` по `id` без проверки тенанта | `WHERE id = $1 AND organization_id = $2` |
| `UpdateConstructionSite` по `id` без проверки тенанта | `WHERE id = $1 AND organization_id = $2` |
| `DeleteConstructionSite` по `id` без проверки тенанта | `WHERE id = $1 AND organization_id = $2` |
| `ListDocumentsBySite` только по `site_id` | `WHERE organization_id = $1 AND site_id = $2` |
| `GetDocument` по `id` без проверки тенанта | `WHERE id = $1 AND organization_id = $2` |
| `DeleteDocument` по `id` без проверки тенанта | `WHERE id = $1 AND organization_id = $2` |
| `GetDocumentTask` без связи с организацией | `JOIN documents ON organization_id = $2` |
| `DeleteDocumentTask` без связи с организацией | subquery `IN (SELECT id FROM documents WHERE organization_id = $2)` |

После изменения SQL-запросов выполнен `make sqlc` — сгенерированы params-struct: `GetConstructionSiteParams`, `DeleteConstructionSiteParams`, `GetDocumentParams`, `DeleteDocumentParams`, `ListDocumentsBySiteParams`, `GetDocumentTaskParams`, `DeleteDocumentTaskParams`.

**5. `DeleteDocument` / `DeleteDocumentTask` → `not_found`**

Оба метода переведены с `:exec` на `:execrows`; сервисы возвращают `errs.CodeNotFound` при `rows == 0` — унификация с паттерном `DeleteConstructionSite`.

**6. `orgIDFromContext` / `userIDFromContext` → методы `Server`**

Пакетные функции конвертированы в методы `(s *Server)`, которые вызывают `s.respondWithError` вместо прямого `c.JSON` — устранено нарушение архитектурного контракта проекта.

**7. Обновление MockStore и исправление `make mock`**

`mockery v2.46.3` несовместима с Go 1.26 (зависит от `golang.org/x/tools@v0.17.0` с багом в константе). Исправлено: `Makefile` переведён на `@latest` (v2.53.6).

После регенерации вручную восстановлена ручная реализация `ExecTx` и тип переименован `Store` → `MockStore` согласно соглашению проекта.

## 2026-04-21 — День 5 (второй раунд): разделение слоёв, DB-уровень защиты, интеграционные тесты

### Что сделано

**1. Разделение документов и задач по слоям (второй раунд code review)**

По итогам PR #5 вынесены в отдельные файлы:

| Слой | До | После |
|---|---|---|
| SQL-запросы | `document.sql` (все) | `document.sql` + `document_task.sql` |
| Repository | `document.sql.go` (все) | `document.sql.go` + `document_task.sql.go` |
| Service | `service_document.go` (все) | `service_document.go` + `service_document_task.go` |
| Handler | `handler_document.go` (все) | `handler_document.go` + `handler_document_task.go` |

Каждый слой теперь отвечает ровно за одну таблицу.

**2. Tenant-изоляция задач**

`ListTasksByDocument` и `UpdateDocumentTaskStatus` добавили `organization_id` через `JOIN documents`:
- `ListTasksByDocument`: `WHERE dt.document_id = $1 AND d.organization_id = $2`
- `UpdateDocumentTaskStatus`: subquery `IN (SELECT id FROM documents WHERE organization_id = $2)`

Сервисный слой и хендлеры обновлены под новые params-struct.

**3. Проверка владения документом в `CreateDocumentTask`**

Перед вставкой задачи хендлер вызывает `documentService.Get(ctx, docID, orgID)` — если документ не принадлежит тенанту, возвращается `404` ещё до INSERT.

**4. Валидация `parent_id` в `CreateConstructionSite`**

Хендлер вызывает `constructionSiteService.Get(ctx, parentID, orgID)` — если родитель из другой организации, возвращается `404` на application-слое (дополнительно к DB-триггеру).

**5. DB-уровень защиты tenant-изоляции (миграция)**

Добавлены в `000001_init_schema.up.sql`:
- `CREATE EXTENSION IF NOT EXISTS pgcrypto`
- `UNIQUE(id, organization_id)` на `users`, `construction_sites`, `documents` — нужны как опора для триггеров
- Триггер `trg_site_org_isolation` (`BEFORE INSERT OR UPDATE ON construction_sites`) — отклоняет `parent_id` из чужой org
- Триггер `trg_document_org_isolation` (`BEFORE INSERT OR UPDATE ON documents`) — отклоняет `site_id` и `parent_id` из чужой org

`000001_init_schema.down.sql` дополнен: дропает триггеры, функции и расширения в правильном порядке.

**6. Mockery зафиксирован**

`Makefile`: `@latest` → `@v2.53.6` — воспроизводимые сборки при `make mock`.

**7. Интеграционные тесты — tenant isolation**

Добавлены 11 новых тестов в `tests/integration/repository_test.go`:

| Тест | Что проверяет |

| `CrossTenantParent_IsRejected` | Триггер блокирует `parent_id` из другого тенанта (sites) |
