# 2026-04-27 — День 10: Extraction keys + сохранение извлечённых данных

## Контекст

Начали слой нормализованного извлечения данных для сравнения контрактов и смет.

Цель: пользователь задаёт вопрос на естественном языке (`Какой процент аванса?`), Go превращает его в стабильный технический ключ, дедуплицирует ключи внутри tenant, передаёт полный список ключей Python extract worker и сохраняет результат воркера в структурированную таблицу.

Текущий Go-normalizer — deterministic fallback (`Какой процент аванса?` → `kakoy_protsent_avansa`). Целевой продуктовый вариант — LLM-based генерация английских snake_case ключей вроде `advance_payment_percent` / `total_amount_m2` с тем же DB/API contract.

## Часть 1 — Схема БД

В `000001_init_schema` добавлены две таблицы.

### `extraction_keys`

Поля:

- `id`
- `organization_id`
- `key_name`
- `source_query`
- `description`
- `data_type`
- `is_required`
- `created_at`, `updated_at`

Ключевой constraint:

```sql
CONSTRAINT uq_extraction_keys_org_key
    UNIQUE (organization_id, key_name)
```

После review `organization_id` сделан `NOT NULL`: текущий extraction flow полностью tenant-scoped, а прежняя идея global keys конфликтовала с composite FK на `document_extracted_data`. Если понадобятся глобальные шаблоны ключей, их лучше вынести в отдельную template-таблицу и копировать в tenant.

### `document_extracted_data`

Поля:

- `id`
- `organization_id`
- `document_id`
- `key_id`
- `extracted_value JSONB`
- `confidence`
- `created_at`, `updated_at`

Tenant isolation сделан на уровне FK:

```sql
CONSTRAINT fk_extracted_data_document_org
    FOREIGN KEY (document_id, organization_id)
    REFERENCES documents(id, organization_id)
```

Для этого в `documents` добавлен `UNIQUE (id, organization_id)`.

Также добавлен FK на `(key_id, organization_id)` → `extraction_keys(id, organization_id)`, чтобы нельзя было привязать результат одного tenant к ключу другого.

После code review unique/upsert ключ усилен до:

```sql
UNIQUE (organization_id, document_id, key_id)
```

Соответственно `UpsertDocumentExtractedData` использует `ON CONFLICT (organization_id, document_id, key_id)`. Это сохраняет DB-level tenant isolation даже при ошибочно переданном `organization_id`.

Для lookup по исходному вопросу добавлен functional index:

```sql
CREATE INDEX idx_extraction_keys_org_norm_source_query
    ON extraction_keys(organization_id, lower(btrim(source_query)));
```

После второго review убран redundant `idx_extracted_data_org_doc`: его покрывает leftmost prefix unique-индекса `(organization_id, document_id, key_id)`.

Индекс `idx_extraction_keys_org` заменён на более полезный для list-запросов:

```sql
CREATE INDEX idx_extraction_keys_org_created_at
    ON extraction_keys(organization_id, created_at);
```

## Часть 3 — API resolve

Добавлен эндпоинт:

```text
POST /api/v1/extraction-keys/resolve
```

Request:

```json
{
  "source_query": "Какой процент аванса?"
}
```

Response:

```json
{
  "key": {
    "id": "...",
    "organization_id": "...",
    "key_name": "kakoy_protsent_avansa",
    "source_query": "Какой процент аванса?",
    "description": "Какой процент аванса?",
    "data_type": "number",
    "is_required": false
  },
  "duplicate": false
}
```

`ExtractionKeyService.Resolve` делает:

1. trim + validation `source_query`;
2. поиск дубля по normalized `source_query`;
3. deterministic генерацию `key_name` через snake_case + Cyrillic transliteration;
4. поиск дубля по `(organization_id, key_name)`;
5. создание нового ключа, если дублей нет.

При unique race сервис повторно читает ключ по `key_name` и возвращает его как duplicate.

## Часть 4 — Extract orchestration через Redis/Celery

Публичное создание задач теперь разрешает:

- `convert`
- `extract`

`anonymize` по-прежнему внутренний модуль: он зависит от markdown-артефакта и создаётся только после завершения `convert`.

Для `extract` `DocumentTaskService.Create` перед INSERT читает ключи tenant:

```go
extractionKeys, err := extractionKeyPayloads(ctx, s.repo, params.OrganizationID)
```

Если SELECT ключей упал, pending-задача не создаётся. Это важно: задача без схемы ключей ушла бы воркеру неполной.

В `pythonworker.ProcessRequest` добавлено поле:

```go
Kwargs map[string]any
```

Celery protocol v2 body теперь:

```json
[
  ["task_id", "document_id", "storage_path"],
  {"extraction_keys": [...]},
  {"callbacks": null, "errbacks": null, "chain": null, "chord": null}
]
```

Для остальных модулей kwargs остаётся `{}`.

## Часть 5 — Callback results

`WorkerService.HandleStatusUpdate` теперь обрабатывает completed `extract`.

Логика:

1. взять `organization_id` документа;
2. загрузить все `extraction_keys` tenant;
3. распарсить JSON результата;
4. сопоставить `key_name` → `key_id`;
5. upsert в `document_extracted_data`.

Поддерживаемые формы payload:

```json
{"extracted_data":[{"key_name":"advance_payment_percent","value":15,"confidence":0.91}]}
```

```json
[{"key_name":"advance_payment_percent","value":15}]
```

```json
{"advance_payment_percent":{"value":15,"confidence":0.91}}
```

```json
{"advance_payment_percent":15}
```

Неизвестные ключи игнорируются. Оригинальный JSON всё равно остаётся в `document_tasks.result_payload`, так что callback можно диагностировать или переиграть.

## Часть 8 — Post-PR review fixes

После публикации PR ревьюеры (Copilot AI + CodeRabbit) обнаружили 4 замечания, все применены.

### rawConfidence: валидация диапазона [0, 1]

`rawConfidence` в `service_worker.go` не проверял, что декодированное число лежит в [0, 1]. Если воркер возвращал, например, `1.5`, функция возвращала `Valid: true`, и `UpsertDocumentExtractedData` падал на DB constraint `confidence BETWEEN 0 AND 1`. Значение не сохранялось, ошибка логировалась, но причина была неочевидна.

Исправление: out-of-range confidence теперь трактуется как отсутствующее (`Valid: false`). Значение при этом всё равно сохраняется без confidence.

Добавлен `TestRawConfidence` (7 кейсов: valid zero, valid one, valid mid, above range, negative, empty, non-number).

### Guard: extract-задача без ключей → validation_failed

`DocumentTaskService.Create` для `module_name="extract"` читает ключи tenant до INSERT. Если ключей нет (`len(extractionKeys) == 0`) — раньше задача всё равно создавалась и уходила воркеру с пустой схемой.

Теперь: `validation_failed` с сообщением `"no extraction keys configured for this organization"`. INSERT не вызывается.

Добавлен `TestDocumentTaskService_Create_ExtractWithNoKeys_ReturnsValidationFailed`.

### Миграция: триггер + индекс

**Триггер:** `extraction_keys` не имел `BEFORE UPDATE OF organization_id` триггера, хотя `users`, `construction_sites` и `documents` имели. Добавлен `trg_extraction_keys_prevent_org_change` — паритет tenant isolation.

**Индекс:** FK `fk_extracted_data_key_org` ссылается на `(key_id, organization_id)`. При CASCADE DELETE из `extraction_keys` Postgres делал scan по `organization_id`. Добавлен `idx_extracted_data_key_org ON document_extracted_data(key_id, organization_id)`.

Down-миграция обновлена (`DROP TRIGGER IF EXISTS trg_extraction_keys_prevent_org_change`).

```bash

make migrate_up
```

Результат: версия `1`, оба объекта присутствуют в схеме.

## Часть 9 — Post-PR review fixes (Round 2)

Второй раунд замечаний Copilot AI — 3 новых правки.

### Миграция: триггер для `document_extracted_data`

`document_extracted_data` содержит `organization_id` FK, однако `BEFORE UPDATE OF organization_id` триггер на ней отсутствовал, несмотря на наличие аналогичных триггеров на `extraction_keys`, `documents`, `construction_sites`, `users`.

Добавлен `trg_document_extracted_data_prevent_org_change`:

```sql
CREATE TRIGGER trg_document_extracted_data_prevent_org_change
    BEFORE UPDATE OF organization_id ON document_extracted_data
    FOR EACH ROW EXECUTE FUNCTION prevent_organization_id_change();
```

Down-миграция дополнена строкой:

```sql
DROP TRIGGER IF EXISTS trg_document_extracted_data_prevent_org_change ON document_extracted_data;
```

### Миграция: удалён избыточный индекс `idx_extracted_data_key_id`

`idx_extracted_data_key_id ON document_extracted_data(key_id)` был полностью покрыт составным `idx_extracted_data_key_org(key_id, organization_id)`. Удалён — составной индекс используется любым запросом по `key_id` (ведущая колонка).

### Комментарий в `service_extraction_key.go`

Комментарий к `transliterateRune` утверждал, что функция "covers Russian/Kazakh Cyrillic", однако в `cyrillicTransliteration` заданы только русские буквы. Комментарий исправлен:

```go
// transliterateRune applies the explicitly defined Cyrillic transliteration
// mapping used to produce stable snake_case fallback keys.
```
