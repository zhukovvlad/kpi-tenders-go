### `internal/service/service_worker.go` — новые методы

#### `triggerExtract(ctx, resolveTask)`

Вызывается из `HandleStatusUpdate` при `module=resolve_keys, status=completed`:

1. Парсит `resolveKeysPayload` из `result_payload`
2. `GetDocumentByID(docID)` — исходный документ
3. `UpsertExtractionKey` для каждого нового ключа из payload (best-effort per-key)
4. `GetDocumentByID(mdDocID)` — MD-артефакт из payload
5. `CreateDocumentTaskInternal(module="extract", inputStoragePath=mdDoc.StoragePath)`
6. `Process(...)` с kwargs `extraction_schema` + `md_document_id`

#### `handleExtractCompleted(ctx, task)`

Вызывается при `module=extract, status=completed`:

1. Парсит `map[string]string` из `result_payload`
2. `GetDocumentByID` — для получения `OrganizationID`
3. `GetExtractionKeysByNames(keyNames, orgID)` — получить ключи по именам
4. `BatchUpsertExtractedData(keyIDs, extractedValues)` — один SQL-запрос через `unnest`

## Часть 3 — PR #16: Code Review фиксы

В рамках ревью PR #16 (ветка `14-semantic-extraction-pipeline`) устранены замечания Copilot AI и CodeRabbit.

### `sql/query/extraction.sql` — детерминированный выбор ключей

`GetExtractionKeysByNames` переписан с `SELECT *` на `SELECT DISTINCT ON (key_name)` с явным `ORDER BY`:

```sql
-- name: GetExtractionKeysByNames :many
SELECT DISTINCT ON (key_name) * FROM extraction_keys
WHERE key_name = ANY(sqlc.arg(key_names)::text[])
  AND (organization_id = sqlc.arg(organization_id)::uuid OR organization_id IS NULL)
ORDER BY key_name, (organization_id IS NULL) ASC;
```

**Проблема до правки**: при наличии и tenant-ключа, и system-ключа с одним именем PostgreSQL мог вернуть любой из них — недетерминированно.

**Решение**: `DISTINCT ON (key_name)` + `ORDER BY key_name, (organization_id IS NULL) ASC` — первым идёт tenant-ключ (`organization_id IS NULL = false = 0`), системный ключ (`= true = 1`) вытесняется.

После изменения: `make sqlc` и `make mock` — оба успешны.

## Часть 4 — PR #16: второй раунд ревью

### `sql/migrations/000002_extraction.up.sql` — trigger для key_id tenant isolation

`document_extracted_data.key_id` имеет FK только на `extraction_keys(id)`, что не мешает вставить ключ другого тенанта в обход составного FK. Добавлен триггер `BEFORE INSERT OR UPDATE`:

```sql
CREATE OR REPLACE FUNCTION trg_check_extracted_data_key_org()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF NOT EXISTS (
        SELECT 1 FROM extraction_keys
        WHERE id = NEW.key_id
          AND (organization_id IS NULL OR organization_id = NEW.organization_id)
    ) THEN
        RAISE EXCEPTION
            'key_id % does not belong to organization % and is not a system key',
            NEW.key_id, NEW.organization_id;
    END IF;
    RETURN NEW;
END;
$$;
```

`down`-миграция обновлена: `DROP TRIGGER` и `DROP FUNCTION` добавлены перед `DROP TABLE`.

### `internal/pythonworker/client.go` — динамический `kwargsrepr`

`kwargsrepr` в Celery-заголовках был жёстко зашит `"{}"` даже при непустых kwargs. Теперь вычисляется через `kwargsRepr(kwargs)` — JSON-строка реальных kwargs (при ошибке сериализации — `"{}"`). Поле информационное, но без него отладка в Celery Inspector затруднена.

### `internal/service/service_extraction.go` — удалён `toOrgUUID`

Неиспользуемый хелпер `toOrgUUID` и импорт `pgtype` удалены.

## Часть 5 — PR #16: третий раунд ревью

### `sql/migrations/000002_extraction.up.sql` — неизменяемость `organization_id`

Добавлены триггеры, блокирующие изменение `organization_id` после вставки на обеих tenant-таблицах:

```sql
CREATE OR REPLACE FUNCTION trg_prevent_org_change()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF NEW.organization_id IS DISTINCT FROM OLD.organization_id THEN
        RAISE EXCEPTION
            'changing organization_id is not allowed on table %', TG_TABLE_NAME;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_immut_org_extraction_keys
    BEFORE UPDATE ON extraction_keys
    FOR EACH ROW EXECUTE FUNCTION trg_prevent_org_change();

CREATE TRIGGER trg_immut_org_document_extracted_data
    BEFORE UPDATE ON document_extracted_data
    FOR EACH ROW EXECUTE FUNCTION trg_prevent_org_change();
```

Индекс `idx_extracted_data_key_org ON document_extracted_data (key_id, organization_id)` добавлен для ускорения trigger-проверки `trg_check_extracted_data_key_org`.

`down`-миграция дополнена `DROP TRIGGER` / `DROP FUNCTION` для новых объектов.

### `sql/query/extraction.sql` + `make sqlc` + `make mock` — `BatchUpsertExtractedData`

Добавлен новый запрос для пакетного upsert всех извлечённых значений одним SQL-оператором:

```sql
-- name: BatchUpsertExtractedData :exec
INSERT INTO document_extracted_data (organization_id, document_id, key_id, extracted_value)
SELECT $1::uuid,
       $2::uuid,
       unnest($3::uuid[]),
       unnest($4::text[])
ON CONFLICT ON CONSTRAINT uq_extracted_data_doc_key DO UPDATE
    SET extracted_value = EXCLUDED.extracted_value;
```

PostgreSQL параллельно разворачивает оба массива (`key_ids`, `extracted_values`) — стандартная техника multi-column unnest.

После добавления запроса: `make sqlc` → `make mock` — оба успешны. Сгенерирована структура:

```go
type BatchUpsertExtractedDataParams struct {
    OrganizationID  uuid.UUID   `json:"organization_id"`
    DocumentID      uuid.UUID   `json:"document_id"`
    KeyIds          []uuid.UUID `json:"key_ids"`
    ExtractedValues []string    `json:"extracted_values"`
}
```

### `internal/service/service_worker.go` — batch upsert в `handleExtractCompleted`

Цикл с N вызовами `UpsertExtractedDatum` заменён одним `BatchUpsertExtractedData`. Это сокращает количество round-trip к БД с O(N) до O(1) на каждый worker-callback:

```go
// Build parallel slices for batch upsert, skipping unknown keys.
keyIDs := make([]uuid.UUID, 0, len(keyIDByName))
extractedValues := make([]string, 0, len(keyIDByName))
for keyName, value := range extractedData {
    keyID, ok := keyIDByName[keyName]
    if !ok {
        s.log.Warn("worker: extract returned unknown key name, skipping",
            "key_name", keyName, "document_id", task.DocumentID)
        continue
    }
    keyIDs = append(keyIDs, keyID)
    extractedValues = append(extractedValues, value)
}
if len(keyIDs) == 0 {
    return nil
}
if err := s.repo.BatchUpsertExtractedData(ctx, repository.BatchUpsertExtractedDataParams{
    OrganizationID:  doc.OrganizationID,
    DocumentID:      task.DocumentID,
    KeyIds:          keyIDs,
    ExtractedValues: extractedValues,
}); err != nil {
    s.log.Error("worker: failed to batch upsert extracted data",
        "document_id", task.DocumentID, "err", err)
}
```

Фильтрация неизвестных ключей перед upsert остаётся: warn-лог и `continue`.

### `internal/service/service_worker_test.go` — обновление тестов под batch

Два теста обновлены под новый интерфейс:

- `ExtractCompleted_UpsertsData`: `ms.On("UpsertExtractedDatum", ...)` → `ms.On("BatchUpsertExtractedData", mock.Anything, mock.MatchedBy(...))` — проверяет `OrganizationID`, `DocumentID`, срезы `KeyIds` и `ExtractedValues`.

## Часть 6 — PR #16: четвёртый раунд ревью

### `sql/query/extraction.sql` — комментарий про unnest

Copilot предложил использовать `FROM unnest(arr1, arr2) AS kv(...)` вместо двух `unnest()` в SELECT-списке. Вариант с FROM более явный, но **sqlc** его не поддерживает (`function unnest(unknown, unknown) does not exist` на этапе анализа). Оставлена текущая форма с двумя `unnest` в SELECT; добавлен комментарий, объясняющий почему это корректно: PostgreSQL с версии 10 гарантирует lockstep-развёртывание нескольких set-returning функций в SELECT-списке.

### `sql/migrations/000002_extraction.up.sql` — расширен UNIQUE в `document_extracted_data`

`uq_extracted_data_doc_key` расширен с `(document_id, key_id)` до `(organization_id, document_id, key_id)`. Это defence-in-depth: `document_id` уже уникально привязан к одной организации через составной FK, поэтому данных дублирования нет, но явная колонка `organization_id` в уникальном ограничении соответствует стандарту tenant isolation проекта. Имя constraint не изменилось → `ON CONFLICT ON CONSTRAINT uq_extracted_data_doc_key` в `UpsertExtractedDatum` и `BatchUpsertExtractedData` не требует правки.

Составной FK `(key_id, organization_id) → extraction_keys(id, organization_id)` **не добавлен**: он несовместим с системными ключами (`extraction_keys.organization_id IS NULL`), при этом триггер `trg_check_extracted_data_key_org` корректно обрабатывает оба случая.

### `sql/migrations/000002_extraction.down.sql` — порядок DROP

Переписан по предложению Copilot. Раньше сначала шли `DROP TRIGGER ... ON <table>` — если таблица отсутствует, PostgreSQL выдаёт ошибку даже с `IF EXISTS` на триггере. Новый порядок:

```sql
DROP TABLE IF EXISTS document_extracted_data CASCADE;  -- убирает все триггеры таблицы
DROP TABLE IF EXISTS extraction_keys CASCADE;
DROP FUNCTION IF EXISTS trg_prevent_org_change();
DROP FUNCTION IF EXISTS trg_check_extracted_data_key_org();
ALTER TABLE documents DROP CONSTRAINT IF EXISTS uq_documents_id_org;
```

`CASCADE` на `DROP TABLE` автоматически удаляет все триггеры таблицы, делая down-миграцию idempotent при частично применённой схеме.

### `internal/service/service_worker.go` — пробрасывание ошибки из `handleExtractCompleted`

CodeRabbit указал, что `BatchUpsertExtractedData` при ошибке логировался и проглатывался (`log.Error` + `return nil`), тогда как `registerConvertArtifacts` и `registerAnonymizeArtifacts` возвращают ошибку, и логирование выполняет вызывающий `HandleStatusUpdate`. Приведено к единому паттерну: функция возвращает `error`, логирует `HandleStatusUpdate` через `s.log.Error("worker: failed to persist extracted data", ...)`.
