# 2026-04-27 — День 10: Extraction keys + сохранение извлечённых данных

## Контекст

Начали слой нормализованного извлечения данных для сравнения контрактов и смет.

Цель: пользователь задаёт вопрос на естественном языке (`Какой процент аванса?`), Go превращает его в стабильный технический ключ (`kakoy_protsent_avansa`), дедуплицирует ключи внутри tenant, передаёт полный список ключей Python extract worker и сохраняет результат воркера в структурированную таблицу.

---

## Часть 1 — Схема БД

В `000001_init_schema` добавлены две таблицы.

### `extraction_keys`

Поля:

- `id`
- `organization_id`
- `key_name`
- `source_query`
- `description`
- `data_type`
- `is_required`
- `created_at`, `updated_at`

Ключевой constraint:

```sql
CONSTRAINT uq_extraction_keys_org_key
    UNIQUE NULLS NOT DISTINCT (organization_id, key_name)
```

`NULLS NOT DISTINCT` оставляет возможность глобальных ключей (`organization_id IS NULL`) и при этом не даёт создать несколько одинаковых глобальных `key_name`.

### `document_extracted_data`

Поля:

- `id`
- `organization_id`
- `document_id`
- `key_id`
- `extracted_value JSONB`
- `confidence`
- `created_at`, `updated_at`

Tenant isolation сделан на уровне FK:

```sql
CONSTRAINT fk_extracted_data_document_org
    FOREIGN KEY (document_id, organization_id)
    REFERENCES documents(id, organization_id)
```

Для этого в `documents` добавлен `UNIQUE (id, organization_id)`.

Также добавлен FK на `(key_id, organization_id)` → `extraction_keys(id, organization_id)`, чтобы нельзя было привязать результат одного tenant к ключу другого.

## Часть 3 — API resolve

Добавлен эндпоинт:

```text
POST /api/v1/extraction-keys/resolve
```

Request:

```json
{
  "source_query": "Какой процент аванса?"
}
```

Response:

```json
{
  "key": {
    "id": "...",
    "organization_id": "...",
    "key_name": "kakoy_protsent_avansa",
    "source_query": "Какой процент аванса?",
    "description": "Какой процент аванса?",
    "data_type": "number",
    "is_required": false
  },
  "duplicate": false
}
```

`ExtractionKeyService.Resolve` делает:

1. trim + validation `source_query`;
2. поиск дубля по normalized `source_query`;
3. deterministic генерацию `key_name` через snake_case + Cyrillic transliteration;
4. поиск дубля по `(organization_id, key_name)`;
5. создание нового ключа, если дублей нет.

При unique race сервис повторно читает ключ по `key_name` и возвращает его как duplicate.

## Часть 4 — Extract orchestration через Redis/Celery

Публичное создание задач теперь разрешает:

- `convert`
- `extract`

`anonymize` по-прежнему внутренний модуль: он зависит от markdown-артефакта и создаётся только после завершения `convert`.

Для `extract` `DocumentTaskService.Create` перед INSERT читает ключи tenant:

```go
extractionKeys, err := extractionKeyPayloads(ctx, s.repo, params.OrganizationID)
```

Если SELECT ключей упал, pending-задача не создаётся. Это важно: задача без схемы ключей ушла бы воркеру неполной.

В `pythonworker.ProcessRequest` добавлено поле:

```go
Kwargs map[string]any
```

Celery protocol v2 body теперь:

```json
[
  ["task_id", "document_id", "storage_path"],
  {"extraction_keys": [...]},
  {"callbacks": null, "errbacks": null, "chain": null, "chord": null}
]
```

Для остальных модулей kwargs остаётся `{}`.

## Часть 5 — Callback results

`WorkerService.HandleStatusUpdate` теперь обрабатывает completed `extract`.

Логика:

1. взять `organization_id` документа;
2. загрузить все `extraction_keys` tenant;
3. распарсить JSON результата;
4. сопоставить `key_name` → `key_id`;
5. upsert в `document_extracted_data`.

Поддерживаемые формы payload:

```json
{"extracted_data":[{"key_name":"advance_payment_percent","value":15,"confidence":0.91}]}
```

```json
[{"key_name":"advance_payment_percent","value":15}]
```

```json
{"advance_payment_percent":{"value":15,"confidence":0.91}}
```

```json
{"advance_payment_percent":15}
```

Неизвестные ключи игнорируются. Оригинальный JSON всё равно остаётся в `document_tasks.result_payload`, так что callback можно диагностировать или переиграть.
