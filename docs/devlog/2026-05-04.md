# 2026-05-04 — Миграция 000004: UI-слой и полный CRUD для новых сущностей

## Контекст

Фронтенд требовал дополнительные сущности: классификаторы документов (виды договоров, роли файлов), приглашения пользователей, лог активности объектов, сессии сравнения документов. Также нужны были новые поля на существующих таблицах: обложка и тип объекта, связи документа с классификаторами, расширенные поля ключей экстракции и пользователей.

## Что сделано

### Миграция `000004_ui_layer`

**Новые таблицы:**

- `document_contract_kinds` — виды договора; `organization_id` nullable (NULL = системный справочник, видимый всем); `UNIQUE NULLS NOT DISTINCT (organization_id, display_name)`.
- `document_file_roles` — роли файла; тот же паттерн, что и `contract_kinds`.
- `user_invitations` — приглашения пользователей; `UNIQUE (organization_id, email) WHERE accepted_at IS NULL`; токен хранится как SHA-256 hash.
- `site_audit_log` — лог событий объекта; INSERT-only, immutability-триггер через `prevent_organization_id_change()`.
- `comparison_sessions` — сессии сравнения; связь с `document_contract_kinds` опциональна.
- `comparison_session_documents` — документы сессии с позицией; `UNIQUE (session_id, document_id)`.

**Новые поля на существующих таблицах:**

- `construction_sites`: `cover_image_path text`, `cover_image_uploaded_at timestamptz`, `site_type text`, `last_activity_at timestamptz NOT NULL DEFAULT now()`.
- `documents`: `contract_kind_id uuid → document_contract_kinds`, `file_role_id uuid → document_file_roles`, `bundle_id uuid` (soft grouping без FK).
- `extraction_keys`: `display_name text`, `is_active bool DEFAULT true`, `category text`.
- `users`: CHECK на `role IN ('admin','member','viewer')`, `last_login_at timestamptz`.

**Вспомогательные объекты:**

- Вью `v_site_status` — агрегирует статус объекта (count задач по статусам, последняя активность, связанные extraction_requests).
- Триггер `propagate_site_activity` — обновляет `last_activity_at` на `construction_sites` при INSERT/UPDATE в `documents` и `document_tasks`.
- Триггер `check_document_kind_role_org` — проверяет, что `contract_kind_id` и `file_role_id` принадлежат той же организации или являются системными (NULL org).
- Immutability-триггеры на новых таблицах.

### Сервисный слой

Новые сервисы:

- **`ContractKindService`** — CRUD; `List` возвращает org-specific + системные записи (NULL org); дублирование по `uq_contract_kinds_org_name` → `CodeConflict`.
- **`FileRoleService`** — аналогичный паттерн.
- **`InvitationService`** — Create хранит только hash токена; raw токен генерируется и возвращается в хендлере; `uq_user_invitations_org_email_active` для дедупликации.
- **`SiteAuditService`** — `Log(ctx, orgID, siteID, actorUserID, eventType, payload)` и `ListBySite` с limit/offset.
- **`ComparisonSessionService`** — CRUD сессий + `AddDocument`/`RemoveDocument`/`ListDocuments`.

Расширения существующих сервисов:

- **`ConstructionSiteService`**: `ListByParent`, `ListRoot`, `UpdateCover`, `UpdateType`.
- **`DocumentService`**: `UpdateMeta(ctx, id, orgID, contractKindID, fileRoleID, bundleID *uuid.UUID)`.

### HTTP-слой

Новые хендлеры и маршруты:

```text
GET/POST         /api/v1/contract-kinds
GET/PATCH/DELETE /api/v1/contract-kinds/:id

GET/POST         /api/v1/file-roles
GET/PATCH/DELETE /api/v1/file-roles/:id

GET/POST         /api/v1/invitations          (AdminOnly)
DELETE           /api/v1/invitations/:id      (AdminOnly)

GET              /api/v1/sites/root
GET              /api/v1/sites/:id/children
PATCH            /api/v1/sites/:id/cover      ({ cover_image_path })
PATCH            /api/v1/sites/:id/type       ({ site_type })
GET              /api/v1/sites/:id/audit-log  (?limit=50&offset=0)

PATCH            /api/v1/documents/:id/meta   ({ contract_kind_id, file_role_id, bundle_id })

GET/POST         /api/v1/comparison-sessions
GET/DELETE       /api/v1/comparison-sessions/:id
POST             /api/v1/comparison-sessions/:id/documents
DELETE           /api/v1/comparison-sessions/:id/documents/:doc_id
```

`POST /api/v1/invitations` генерирует 32-байтный случайный токен, возвращает `{invitation, token}` — raw токен в БД не хранится, только `sha256(token)`.

## Ловушки

- **`ListContractKindsByOrg` принимает `uuid.UUID`, не `pgtype.UUID`** — sqlc сгенерировал сигнатуру с `dollar_1 uuid.UUID` из-за `$1::uuid` каста в WHERE. Аналогично `GetContractKindParams.Column2` вместо `OrganizationID` — sqlc не инферирует имя из выражения `$2::uuid`.
- **`CreateUserInvitationParams.ExpiresAt` — `time.Time`**, несмотря на `timestamptz` в схеме: pgx v5 + sqlc маппят `timestamptz` в `time.Time` напрямую без `pgtype.Timestamptz`.
- **Порядок маршрутов Gin:** `GET /sites/root` должен регистрироваться **до** `GET /sites/:id`, иначе Gin матчит `/root` как параметр `:id`.
