# 2026-05-05 — Code Review PR #17: исправление багов и улучшение качества

## Контекст

После мёрджа PR #17 (миграция 000004 + UI CRUD) Copilot и coderabbitai выявили ряд дефектов: проблемы безопасности (tenant isolation, PII в логах, просроченные инвайты), нестабильная пагинация, silent state corruption (PATCH-хендлеры затирали поля нулями), баги в state machine ExtractionService.

## Что сделано

### Исправления SQL-запросов (7 мест)

**1. Tenant isolation в `RemoveDocumentFromComparisonSession`**

```sql
-- Было: не проверяло принадлежность документа организации
DELETE FROM comparison_session_documents WHERE session_id = $1 AND document_id = $2;

-- Стало: добавлен org-фильтр через sqlc.arg
DELETE FROM comparison_session_documents
WHERE session_id = sqlc.arg(session_id)
  AND document_id = sqlc.arg(document_id)
  AND organization_id = sqlc.arg(organization_id);
```

Сервис (`RemoveDocument`) получил новый параметр `orgID uuid.UUID`; хендлер передаёт его из контекста.

**2. Просроченные инвайты в `AcceptUserInvitation`**

```sql
-- Было: принимало любой неиспользованный инвайт
WHERE id = $1 AND accepted_at IS NULL

-- Стало:
WHERE id = $1 AND accepted_at IS NULL AND expires_at > now()
```

**3. Условный `cover_image_uploaded_at` в `UpdateConstructionSiteCover`**

```sql
-- Было: всегда обновляло timestamp, даже при удалении обложки
cover_image_uploaded_at = now()

-- Стало: обновляет только при установке нового пути
cover_image_uploaded_at = CASE WHEN $3 IS NOT NULL THEN now() ELSE cover_image_uploaded_at END
```

**4. Детерминированная пагинация в `ListSiteAuditLogBySite`**

```sql
-- Было: ORDER BY created_at DESC — нестабильно при одинаковом timestamp
ORDER BY created_at DESC, id DESC
```

**5–6. sqlc.arg вместо `$N::uuid` в `contract_kind.sql` и `file_role.sql`**

`$1::uuid` и `$2::uuid` генерировали поля `dollar_1 uuid.UUID` и `Column2 uuid.UUID` в структурах. Заменены на `sqlc.arg(organization_id)` → читаемые `OrganizationID pgtype.UUID`. Сервисы обновлены: `Column2: orgID` → `OrganizationID: pgtype.UUID{Bytes: orgID, Valid: true}`.

**7. PATCH-семантика в `UpdateDocumentMeta` (COALESCE + parent_id guard)**

```sql
-- Было: отправка NULL затирала поле
SET contract_kind_id = $3, file_role_id = $4, bundle_id = $5

-- Стало: опущенные поля сохраняют существующее значение
SET contract_kind_id = COALESCE($3, contract_kind_id),
    file_role_id     = COALESCE($4, file_role_id),
    bundle_id        = COALESCE($5, bundle_id),
    updated_at       = now()
WHERE id = $1 AND organization_id = $2
  AND parent_id IS NULL   -- артефакты не могут быть обновлены
```

После всех SQL-правок: `make sqlc` + `make mock`.

### Исправления в state machine

**`service_worker.go` — pending extraction_requests зависают при ошибке convert/anonymize**

При `status=failed` на convert или anonymize все связанные `extraction_requests` в статусе `pending` оставались висеть. Добавлен новый вспомогательный метод:

```go
func (s *WorkerService) failPendingRequests(ctx context.Context, documentID uuid.UUID, reason string)
```

Вызывается в `HandleStatusUpdate` при `status=failed && moduleName in (convert, anonymize)`. Обходит `ListPendingExtractionRequestsByDocument` и для каждого зовёт `pipeline.MarkRequestFailed`. Best-effort: ошибки логируются, цикл продолжается.

**`service_extraction.go` — extraction_request зависает при ошибке публикации в Redis**

Если `pythonClient.Process` падал при `enqueueResolveKeys` или `enqueueExtract`, задача помечалась failed, но соответствующий `extraction_request` оставался в статусе `running`. Добавлен вызов `MarkRequestFailed` после `markTaskFailed`:

```go
s.markTaskFailed(ctx, task.ID, err.Error())
s.MarkRequestFailed(ctx, req.ID, "failed to publish resolve_keys task")
```

## Rounds 9–10: дополнительные исправления по ревью

### Round 9 — три независимых бага

**C1: `OnExtractCompleted` не detach-ил контекст**

`OnResolveKeysCompleted` внутри создавал `chainCtx` с `context.WithoutCancel + pipelineTimeout`. `OnExtractCompleted` этого не делал — все DB-операции (persist extracted data + mark completed) выполнялись на исходном HTTP-request context'е воркера. При disconnect воркера в середине вызова данные не сохранялись.

Исправление (`service_extraction.go`): добавлен `chainCtx, cancel := context.WithTimeout(context.WithoutCancel(ctx), pipelineTimeout)` в начало `OnExtractCompleted` — все последующие вызовы репозитория идут через `chainCtx`.

**C2/C3: nil guards для cover и site_type блокировали легитимные NULL-запросы**

В Round 8 в хендлеры `UpdateConstructionSiteCover` и `UpdateConstructionSiteType` были добавлены nil-guards, возвращавшие 400 при отсутствии поля. Однако SQL специально поддерживает NULL:

```sql
-- UpdateConstructionSiteCover:
cover_image_uploaded_at = CASE WHEN $3 IS NOT NULL THEN now() ELSE cover_image_uploaded_at END
```

Nil (отправка `"cover_image_path": null`) — это легитимная операция очистки обложки. Nil guards убраны из обоих хендлеров.

**C4: тесты для `handler_comparison_session.go`**

После tenant-scoped изменения `ListDocuments(ctx, sessionID, orgID)` в Round 8 хендлер не имел ни одного теста. Создан `handler_comparison_session_test.go` — 19 кейсов:

- `ListComparisonSessions`: 401, success, 500 DB error
- `GetComparisonSession`: 401, 400 невалидный UUID, 404, success (проверяет что `ListDocuments` вызывается с `orgID`)
- `CreateComparisonSession`: 401, 400 невалидный `contract_kind_id`, success
- `DeleteComparisonSession`: 401, 404 (0 строк), success 204
- `AddDocumentToSession`: 401, 404 сессия не найдена, success 201
- `RemoveDocumentFromSession`: 401, 404 (0 строк), success 204

### Round 10: `registerConvertArtifacts` должен быть безусловным

**Баг:** при ошибке `triggerAnonymize` (Redis недоступен, `CreateDocumentTaskSingleton` упал) `registerConvertArtifacts` пропускался — `convert_md` Document в БД не создавался. `ExtractionService.Progress` обнаруживает артефакты через `ListDocumentsByParent`: без `convert_md`-записи любой `anonymize=false` `extraction_request` зависает навсегда.

**Исправление (`service_worker.go`):** изменён порядок в блоке `convert completed`:

```go
// Было (artifact only if anonymize succeeds):
if err := s.triggerAnonymize(...); err != nil {
    log.Error(...)
} else {
    registerConvertArtifacts(...)
}

// Стало (artifact always first):
if err := runWithArtifactTimeout(ctx, task, s.registerConvertArtifacts); err != nil {
    log.Error(...)
}
if err := s.triggerAnonymize(ctx, task); err != nil {
    log.Error(...)
}
```

`triggerAnonymize` по-прежнему читает `md_storage_path` из in-memory `task` (значение, возвращённое `UpdateWorkerTaskStatus`), который `registerConvertArtifacts` не изменяет — она только пишет в БД. Оба шага best-effort: ошибки логируются.

Обновлён тест `TestWorkerService_HandleStatusUpdate_PythonClientError_NoErrorPropagated`: теперь ожидает `GetDocumentByID + CreateArtifactDocument + UpdateTaskResultPayload` до `Process` (т.е. артефакт регистрируется даже когда Redis падает).

## Rounds 11–12: финальные исправления по ревью PR #17

### Round 11 — четыре точечных фикса

**C1: Имя constraint в `IsUniqueViolation` (`service_document_task.go`)**

После миграции 000003 constraint `uq_document_tasks_document_module` был удалён и заменён двумя partial-индексами (`uq_document_tasks_doc_singleton`, `uq_document_tasks_request_module`). Код сравнивал со старым именем → конфликт никогда не отлавливался, документ-таска создавалась повторно.

```go
// Было:
pgutil.IsUniqueViolation(err, "uq_document_tasks_document_module")

// Стало:
pgutil.IsUniqueViolation(err, "uq_document_tasks_doc_singleton")
```

**C2: Silent PATCH clobber для `cover_image_path` и `site_type`**

`UpdateConstructionSiteCover` и `UpdateConstructionSiteType` использовали `ShouldBindJSON`, не различая `{}` (поле отсутствует) и `{"cover_image_path": null}` (явная очистка). Любой `{}` затирал поле нулём.

Решение — `io.ReadAll + json.Unmarshal` в raw map с проверкой наличия ключа:

```go
raw, _ := io.ReadAll(c.Request.Body)
var check map[string]any
json.Unmarshal(raw, &check)
if _, ok := check["cover_image_path"]; !ok {
    s.respondWithError(c, errs.New(errs.CodeValidationFailed, "cover_image_path is required", nil))
    return
}
```

`{"cover_image_path": null}` по-прежнему проходит и очищает поле (легитимная операция).

**C3: Down-миграция 000003 — `ADD CONSTRAINT` падал при наличии extract/resolve_keys строк**

```sql
-- Добавлено перед ADD CONSTRAINT:
DELETE FROM document_tasks
WHERE extraction_request_id IS NOT NULL;
-- Деструктивно по дизайну: down-миграция удаляет extraction_request_id из схемы.
```

**C4: TrimSpace на query-параметрах в `handler_document_task.go`**

`document_ids=" "` проходил пустую строку, split давал непустой `[""]`, UUID parse возвращал мусорную ошибку вместо 400 "parameter required".

```go
docIDsStr := strings.TrimSpace(c.Query("document_ids"))
// ...
rawParts := strings.Split(docIDsStr, ",")
parts := make([]string, 0, len(rawParts))
for _, rp := range rawParts {
    if strings.TrimSpace(rp) != "" {
        parts = append(parts, strings.TrimSpace(rp))
    }
}
```
