# Дневник разработки — go-kpi-tenders

## 2026-04-24 — День 8: Worker callback + task chaining (convert → anonymize)

### Контекст

Python-воркер умеет выполнять задачи и возвращать результат, но Go-сторона не имела эндпоинта для приёма callback'ов. Также нужна цепочка модулей: когда `convert` завершается, Go должен автоматически создать задачу `anonymize` и триггернуть Python.

---

### Что сделано

**1. SQL-запросы — `sql/query/document_task.sql`**

Добавлены два внутренних запроса без org-check:

```sql
-- name: UpdateWorkerTaskStatus :one
UPDATE document_tasks
SET
    status         = $2,
    celery_task_id = COALESCE($3, celery_task_id),
    result_payload = COALESCE($4, result_payload),
    error_message  = COALESCE($5, error_message),
    updated_at     = now()
WHERE id = $1
RETURNING *;

-- name: CreateDocumentTaskInternal :one
INSERT INTO document_tasks (document_id, module_name)
VALUES ($1, $2)
ON CONFLICT (document_id, module_name) DO NOTHING
RETURNING *;
```

`ON CONFLICT DO NOTHING` делает запрос идемпотентным: повторный вызов для той же пары `(document_id, module_name)` не создаёт дубликат и возвращает `pgx.ErrNoRows`, который caller трактует как «задача уже существует».

Оба запроса — **internal-only**: org-check намеренно отсутствует, защита идёт через `ServiceBearerAuth` middleware. Существующий `UpdateDocumentTaskStatus` (с org-check) не тронут — он остаётся для публичного API.

После добавления: `make sqlc` + `make mock`.

**2. Пакет `internal/pythonworker`**

Новый пакет-клиент для HTTP-вызовов к Python-сервису:

```go
type Client struct {
    baseURL    string
    httpClient *http.Client  // no client-level timeout; deadline managed by ctx
}

func (c *Client) Process(ctx context.Context, req ProcessRequest) error
```

URL настраивается через `PYTHON_SERVICE_URL`. Авторизация не требуется — Python-сервис недоступен снаружи сети. `WorkerService` принимает не `*Client` напрямую, а consumer-side interface `workerPythonClient` — это стандартный Go-идиом, позволяющий мокировать клиент в тестах. Если `PYTHON_SERVICE_URL` не задан, `WorkerService` не инициализируется и callback-эндпоинт возвращает 500.

**3. Config — `internal/config/config.go`**

```go
PythonServiceURL string `env:"PYTHON_SERVICE_URL"`
```

Поле **опциональное** — значение по умолчанию отсутствует. Пустая строка означает, что worker-интеграция отключена: `WorkerService` не инициализируется в `NewServer()`. Если значение задано, `validate()` проверяет, что это абсолютный `http`/`https` URL с хостом и без фрагмента.

**4. `WorkerService` — `internal/service/service_worker.go`**

Сервис принимает `repository.Querier` и `workerPythonClient`. Основная логика:

```
HandleStatusUpdate(taskID, upd WorkerStatusUpdate):
  1. UpdateWorkerTaskStatus → persist callback
  2. if module_name == "convert" && status == "completed":
       → CreateDocumentTaskInternal("anonymize")
       → pythonClient.Process(...)
     // ошибка шага 2 только логируется, наружу не пробрасывается
```

Решение по chaining: вместо денормализации `organization_id` в `document_tasks` добавлен `CreateDocumentTaskInternal` — запрос без org-check, схему БД не меняет.

**5. Хендлер `WorkerUpdateTaskStatus`**

```
PATCH /internal/worker/tasks/:id/status
```

Защищён `ServiceBearerAuth()`. Валидирует `status` перед вызовом сервиса (допустимые значения: `processing`, `completed`, `failed`).

**6. Инициализация в `Server`**

`workerService *service.WorkerService` добавлено как поле `Server`. В `NewServer()`:

```go
workerService: service.NewWorkerService(db, pythonworker.New(cfg.PythonServiceURL), log),
```

Роутер: убрана заглушка `/ping`, добавлен реальный маршрут:

```go
internal.PATCH("/tasks/:id/status", s.WorkerUpdateTaskStatus)
```

**5. Документация в `handler_worker.go`**

Добавлен комментарий поज़вольному whitelist статусов (почему `pending` не включён).

**6. Отклоненные замечания**

- **COALESCE вместо CASE WHEN** для `error_message` — оставлено. `failed` является финальным статусом; при retry создаётся новая задача. Проблема с stale ошибкой неактуальна.

## Консолидация миграций (24 апреля)

### Контекст

Миграция `000002` добавляла UNIQUE constraint на `document_tasks(document_id, module_name)` через `ALTER TABLE`. Исторически это правильно для production (нельзя изменить уже применённую миграцию), но в рамках этого проекта БД ещё не деплоилась в production — «кривая история» нецелесообразна.

### Что сделано

UNIQUE constraint перенесён прямо в `CREATE TABLE document_tasks` в `000001_init_schema.up.sql`:

```sql
CREATE TABLE document_tasks (
    ...
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_document_tasks_document_module UNIQUE (document_id, module_name)
);
```

Файлы `000002_document_tasks_unique_module.up.sql` и `*.down.sql` удалены. БД сброшена и поднята заново через одну миграцию.

**Процедура сброса (миграция помнила версию 2):**

```bash
migrate ... force 1   # откатить версию в schema_migrations на 1
migrate ... down 1    # применить 000001.down.sql (DROP TABLE …)
migrate ... up        # применить 000001.up.sql с inline UNIQUE
```

### Результат

```
UNIQUE CONSTRAINT, btree (document_id, module_name) — uq_document_tasks_document_module
```

Присутствует прямо в `CREATE TABLE`, история миграций: одна версия `000001`. Следующая миграция для `catalog_positions` (pgvector RAG) будет `000002`.

### Контекст

Copilot code review после консолидации миграций. 2 замечания: UNIQUE-конфликт возвращал 500 вместо 409, и chaining в `triggerAnonymize` использовал request-контекст.

### Что сделано

**1. UNIQUE-нарушение → 409 Conflict в `DocumentTaskService`**

`service_document_task.go` возвращал `CodeInternalError` на любую неизвестную ошибку. Добавлена проверка:

```go
if pgutil.IsUniqueViolation(err, "uq_document_tasks_document_module") {
    return ..., errs.New(errs.CodeConflict, "task for this module already exists", err)
}
```

**2. `context.WithoutCancel` в `triggerAnonymize`**

Chaining-работа (создание anonymize-задачи + вызов Python) выполнялась в request-контексте. Если воркер отключался до завершения — задача зависала в `pending`. Исправлено: отвязка от request + 30-секундный таймаут:

```go
chainCtx, cancel := context.WithTimeout(context.WithoutCancel(ctx), 30*time.Second)
defer cancel()
```
