# Дневник разработки — go-kpi-tenders

## 2026-04-24 — День 8: Worker callback + task chaining (convert → anonymize)

### Контекст

Python-воркер умеет выполнять задачи и возвращать результат, но Go-сторона не имела эндпоинта для приёма callback'ов. Также нужна цепочка модулей: когда `convert` завершается, Go должен автоматически создать задачу `anonymize` и триггернуть Python.

---

### Что сделано

**1. SQL-запросы — `sql/query/document_task.sql`**

Добавлены два внутренних запроса без org-check:

```sql
-- name: UpdateWorkerTaskStatus :one
UPDATE document_tasks
SET
    status         = $2,
    celery_task_id = COALESCE($3, celery_task_id),
    result_payload = COALESCE($4, result_payload),
    error_message  = COALESCE($5, error_message),
    updated_at     = now()
WHERE id = $1
RETURNING *;

-- name: CreateDocumentTaskInternal :one
INSERT INTO document_tasks (document_id, module_name)
VALUES ($1, $2)
RETURNING *;
```

Оба запроса — **internal-only**: org-check намеренно отсутствует, защита идёт через `ServiceBearerAuth` middleware. Существующий `UpdateDocumentTaskStatus` (с org-check) не тронут — он остаётся для публичного API.

После добавления: `make sqlc` + `make mock`.

**2. Пакет `internal/pythonworker`**

Новый пакет-клиент для HTTP-вызовов к Python-сервису:

```go
type Client struct {
    baseURL    string
    httpClient *http.Client  // Timeout: 10s
}

func (c *Client) Process(ctx context.Context, req ProcessRequest) error
```

URL настраивается через `PYTHON_SERVICE_URL`. Авторизация не требуется — Python-сервис недоступен снаружи сети. `WorkerService` принимает не `*Client` напрямую, а consumer-side interface `workerPythonClient` — это стандартный Go-идиом, позволяющий мокировать клиент в тестах.

**3. Config — `internal/config/config.go`**

```go
PythonServiceURL string `env:"PYTHON_SERVICE_URL" env-default:"http://localhost:8000"`
```

**4. `WorkerService` — `internal/service/service_worker.go`**

Сервис принимает `repository.Querier` и `workerPythonClient`. Основная логика:

```
HandleStatusUpdate(taskID, upd WorkerStatusUpdate):
  1. UpdateWorkerTaskStatus → persist callback
  2. if module_name == "convert" && status == "completed":
       → CreateDocumentTaskInternal("anonymize")
       → pythonClient.Process(...)
     // ошибка шага 2 только логируется, наружу не пробрасывается
```

Решение по chaining: вместо денормализации `organization_id` в `document_tasks` добавлен `CreateDocumentTaskInternal` — запрос без org-check, схему БД не меняет.

**5. Хендлер `WorkerUpdateTaskStatus`**

```
PATCH /internal/worker/tasks/:id/status
```

Защищён `ServiceBearerAuth()`. Валидирует `status` перед вызовом сервиса (допустимые значения: `processing`, `completed`, `failed`).

**6. Инициализация в `Server`**

`workerService *service.WorkerService` добавлено как поле `Server`. В `NewServer()`:

```go
workerService: service.NewWorkerService(db, pythonworker.New(cfg.PythonServiceURL), log),
```

Роутер: убрана заглушка `/ping`, добавлен реальный маршрут:

```go
internal.PATCH("/tasks/:id/status", s.WorkerUpdateTaskStatus)
```

### Стратегия тестирования

**`internal/service/service_worker_test.go`** — 6 unit-тестов с `mock.MockStore` и `mockPythonClient`:

| Тест | Сценарий |
|------|----------|
| `TestWorkerService_HandleStatusUpdate_Processing` | `status=processing` → только UPDATE, chaining не запускается |
| `TestWorkerService_HandleStatusUpdate_ConvertCompleted_TriggersAnonymize` | `module=convert, status=completed` → UPDATE + `CreateDocumentTaskInternal` + `pythonClient.Process` |
| `TestWorkerService_HandleStatusUpdate_AnonymizeCompleted_NoChaining` | `module=anonymize, status=completed` → только UPDATE |
| `TestWorkerService_HandleStatusUpdate_Failed` | `status=failed` → UPDATE с `error_message`, chaining не запускается |
| `TestWorkerService_HandleStatusUpdate_PythonClientError_NoErrorPropagated` | `Process` возвращает ошибку → `HandleStatusUpdate` возвращает `nil` (ошибка залоггирована) |
| `TestWorkerService_HandleStatusUpdate_TaskNotFound` | `pgx.ErrNoRows` → `errs.CodeNotFound` |

`mockPythonClient` — hand-written testify-mock для `workerPythonClient` interface, определён в самом тест-файле.

**Обновлены `middleware_test.go`**: четыре теста `ServiceBearerAuth` переключены с удалённого `/ping` на реальный `PATCH /internal/worker/tasks/:id/status`.
