## Часть 1 — Stale Task Watchdog

Реализован сторожевой таймер для зависших задач (`document_tasks` со статусом `processing`).

### Миграция 000002

**`sql/migrations/000002_stale_task_retry_count.up.sql`**

```sql
ALTER TABLE document_tasks
    ADD COLUMN IF NOT EXISTS retry_count INT NOT NULL DEFAULT 0;

CREATE INDEX IF NOT EXISTS idx_document_tasks_stale
    ON document_tasks (updated_at ASC)
    WHERE status = 'processing';
```

- `retry_count` — счётчик повторных попыток (по умолчанию 0).
- Partial index ускоряет выборку только «живых» processing-задач без full scan.

---

### SQL-запросы (`sql/query/document_task.sql`)

Добавлены три новых запроса:

```sql
-- name: ListStaleTasks :many
SELECT dt.id, dt.document_id, dt.module_name, dt.retry_count, d.storage_path
FROM document_tasks dt JOIN documents d ON d.id = dt.document_id
WHERE dt.status = 'processing' AND dt.updated_at < $1
ORDER BY dt.updated_at ASC;

-- name: MarkStaleTaskPending :execrows
UPDATE document_tasks SET status = 'pending', retry_count = retry_count + 1, updated_at = now()
WHERE id = $1 AND status = 'processing';

-- name: MarkStaleTaskFailed :execrows
UPDATE document_tasks SET status = 'failed',
    error_message = 'stale task: exceeded max retry attempts', updated_at = now()
WHERE id = $1 AND status = 'processing';
```

`MarkStaleTaskPending` и `MarkStaleTaskFailed` используют `WHERE status = 'processing'` как CAS: если другой worker уже забрал задачу, `rowsAffected = 0` и watchdog молча пропускает.

Также `GetDocumentTask` и `ListTasksByDocument` переведены на `dt.*` вместо явного перечисления колонок — это предотвращает генерацию sqlc отдельных `*Row`-типов при JOIN, что ломало бы service-код.

---

### `internal/watchdog`

### Дизайн

```go
type Watchdog struct {
    q   repository.Querier
    pub pythonPublisher   // consumer-side interface: только Process()
    cfg *config.Config
    log *slog.Logger
}
```

- `Start(ctx)` — блокирующий цикл на `time.Ticker` с интервалом `cfg.WatchdogInterval`.
- `runOnce(ctx)` — основная логика: выборка задач старше `cfg.WatchdogThreshold`, обработка каждой.
- `processTask()` — если `RetryCount >= MaxRetries` → `MarkStaleTaskFailed`; иначе → `MarkStaleTaskPending` + `Process()`.
- Ошибки публикации в Redis логируются и не пробрасываются (best-effort).

### Конфиг

```env
WATCHDOG_INTERVAL    = 2m    # частота проверок
WATCHDOG_THRESHOLD   = 10m   # задача считается зависшей
WATCHDOG_MAX_RETRIES = 5     # после этого — permanent fail
```

---

### Подключение в `cmd/api/main.go`

```go
watchdogCtx, watchdogCancel := context.WithCancel(context.Background())
defer watchdogCancel()
go watchdog.Start(watchdogCtx, srv.DB(), srv.PythonPublisher(), cfg, log)
```

Добавлены геттеры `Server.DB()` и `Server.PythonPublisher()` для передачи зависимостей без нарушения инкапсуляции.

## Часть 3 — PR #14 code review: 5 замечаний

Copilot code review на PR #14 выдал 5 замечаний. Часть касалась корректности SQL (race condition), часть — защитного программирования (`fileNameFromPath`, валидация конфига), часть — полноты схемы (`ON CONFLICT` для артефактов). После закрытия всех замечаний приняли решение сквошить все 5 накопившихся миграций в одну.

---

### 1. TOCTOU-фикс в watchdog SQL

`MarkStaleTaskPending` и `MarkStaleTaskFailed` обновляли задачу только по `id`. Два конкурирующих экземпляра watchdog могли захватить одну задачу одновременно: оба видят её в `ListStaleTasks`, оба делают UPDATE, оба публикуют в Redis — задача выполняется дважды.

Исправление: `AND updated_at < $2` в оба UPDATE. Параметр — `cutoff` (тот же timestamp, что и в `ListStaleTasks`). UPDATE атомарно проверяет, что задача не изменилась после листинга:

```sql
UPDATE document_tasks
SET    status = 'pending', retry_count = retry_count + 1, updated_at = now()
WHERE  id = $1
  AND  updated_at < $2
RETURNING *;
```

Второй конкурент получит 0 строк → `pgx.ErrNoRows` → CAS-пропуск.

`processTask` расширена сигнатурой `cutoff time.Time`; `runOnce` вычисляет и передаёт `cutoff` в оба вызова.

**Тесты**: 5 тестов обновлены под `MarkStale*Params` через `mock.MatchedBy(func(p ...) bool { return p.ID == taskID })` — нечувствительно к конкретному значению `cutoff`.

### 2. `CreateArtifactDocument` — `DO UPDATE SET` вместо `DO NOTHING`

При повторном callback воркера (retry, idempotent replay) файл в MinIO мог обновиться, а запись в БД оставалась старой — `DO NOTHING` молча игнорировал новые данные.

Исправлено на:

```sql
ON CONFLICT (parent_id, artifact_kind) WHERE parent_id IS NOT NULL
DO UPDATE SET
    file_name       = EXCLUDED.file_name,
    storage_path    = EXCLUDED.storage_path,
    mime_type       = EXCLUDED.mime_type,
    file_size_bytes = EXCLUDED.file_size_bytes,
    updated_at      = now();
```

### 3. Валидация watchdog-конфига

`validate()` не проверяла поля Watchdog. `WATCHDOG_INTERVAL=0` → спиннинг; `WATCHDOG_THRESHOLD=-1m` → все задачи считаются зависшими немедленно.

Добавлены три проверки:

```go
if c.WatchdogInterval <= 0 { return fmt.Errorf("WATCHDOG_INTERVAL must be positive (got %s)", ...) }
if c.WatchdogThreshold <= 0 { return fmt.Errorf("WATCHDOG_THRESHOLD must be positive (got %s)", ...) }
if c.WatchdogMaxRetries < 0 { return fmt.Errorf("WATCHDOG_MAX_RETRIES must be >= 0 (got %d)", ...) }
```

### 4. `fileNameFromPath` → `path.Base`

`strings.Split("bucket/docs/", "/")` → последний элемент `""`. Для MinIO-путей с trailing slash функция возвращала пустую строку.

```go
// было
parts := strings.Split(storagePath, "/")
return parts[len(parts)-1]

// стало
func fileNameFromPath(storagePath string) string {
    if storagePath == "" {
        return ""
    }
    return path.Base(storagePath)
}
```

`path.Base` trim'ит trailing slashes перед извлечением имени.

### 5. Бэкфилл `input_storage_path` в миграции

Колонка добавлялась с `DEFAULT ''`, но уже существующие `convert`-задачи имели реальные пути к файлам. Watchdog при перезапуске таких задач отправлял воркеру пустой путь.

Добавлен бэкфилл: UPDATE через JOIN для `convert`, затем fallback для остальных задач с непустым `storage_path`.

### 6. `parent_id` убран из публичного `POST /api/v1/documents`

CHECK constraint `documents_parent_artifact_kind_chk` требует, чтобы при `parent_id IS NOT NULL` поле `artifact_kind` тоже было задано. Публичный хендлер `CreateDocument` принимал `parent_id`, но `artifact_kind` в запросе не было — любой `parent_id` в теле запроса приводил бы к нарушению constraint на INSERT.

Решение (вариант a): `parent_id` убран из `createDocumentRequest`. Артефакты создаёт только воркер через `CreateArtifactDocument` — с явным `artifact_kind`. Публичный эндпоинт создаёт исключительно оригинальные документы (`parent_id IS NULL`, `artifact_kind IS NULL`).

### 7. Retry-loop при публикации в watchdog

`MarkStaleTaskPending` обновляет `updated_at = now()` до попытки отправки в Redis. Если Redis недоступен — задача остаётся в `pending` с "свежим" timestamp'ом и не будет считаться зависшей до следующего цикла (через `WatchdogThreshold`).

Добавлен retry-loop: 3 попытки с 500ms паузой между ними. После исчерпания попыток — best-effort логирование, задача гарантированно будет перезапущена на следующем тике watchdog'а.

```go
const maxPublishAttempts = 3
const publishAttemptTimeout = 5 * time.Second
const publishRetryDelay = 500 * time.Millisecond
```

Тест `TestRunOnce_BestEffortOnPublishError` обновлён: `.Times(3)` — ожидает ровно 3 вызова `Process`.

### 8. `sync.WaitGroup` для корректного shutdown watchdog-горутины

`watchdogCancel()` отменяет контекст горутины, но не ждёт её завершения. Если `srv.Close()` (закрывает пул Redis-соединений) вызывается раньше, чем watchdog выходит из текущего `processTask`, возникает race: in-flight `LPUSH` использует уже закрытый пул.

```go
var watchdogDone sync.WaitGroup
watchdogDone.Add(1)
go func() {
    defer watchdogDone.Done()
    watchdog.Start(watchdogCtx, srv.DB(), srv.PythonPublisher(), cfg, log)
}()

// shutdown:
watchdogCancel()
watchdogDone.Wait() // горутина завершена → Redis закрывать безопасно
```

Подпись `watchdog.Start` не менялась — `WaitGroup` инкапсулирован в `cmd/api/main.go`. В импорты добавлен `"sync"`.

### 9. Константа `moduleConvert` вместо хардкода в `service_document_task.go`

`DocumentTaskService.Create` проверял `params.ModuleName != "convert"` — строковый литерал. Константа `moduleConvert = "convert"` уже была объявлена в `service_worker.go` (тот же пакет `service`).

```go
// было
if params.ModuleName != "convert" {
    return ..., errs.New(..., "only 'convert' is allowed", nil)
}

// стало
if params.ModuleName != moduleConvert {
    return ..., errs.New(..., fmt.Sprintf("only %q is allowed", moduleConvert), nil)
}
```

Сообщение об ошибке также использует константу через `%q` — при переименовании модуля текст ошибки обновится автоматически.

### 10. Покрытие хендлера для ветки `?parent_id=`

Copilot отметил, что `ListDocuments` содержит новую ветку (`parent_id` query-параметр) без единого теста. Добавлено три кейса в `handler_document_test.go`:

| Тест | Сценарий | HTTP |
|------|----------|------|
| `TestListDocuments_ByParentID_Success` | valid UUID → `GetDocument` (ownership check) + `ListDocumentsByParent` → 200 + массив артефактов | 200 |
| `TestListDocuments_InvalidParentID_Returns400` | `parent_id=not-a-uuid` → uuid.Parse падает → validation_failed | 400 |
