# 2026-04-26 — Stale Task Watchdog

## Что сделано

Реализован сторожевой таймер для зависших задач (`document_tasks` со статусом `processing`).

---

## Миграция 000002

**`sql/migrations/000002_stale_task_retry_count.up.sql`**

```sql
ALTER TABLE document_tasks
    ADD COLUMN IF NOT EXISTS retry_count INT NOT NULL DEFAULT 0;

CREATE INDEX IF NOT EXISTS idx_document_tasks_stale
    ON document_tasks (updated_at ASC)
    WHERE status = 'processing';
```

- `retry_count` — счётчик повторных попыток (по умолчанию 0).
- Partial index ускоряет выборку только «живых» processing-задач без full scan.

---

## SQL-запросы (`sql/query/document_task.sql`)

Добавлены три новых запроса:

```sql
-- name: ListStaleTasks :many
SELECT dt.id, dt.document_id, dt.module_name, dt.retry_count, d.storage_path
FROM document_tasks dt JOIN documents d ON d.id = dt.document_id
WHERE dt.status = 'processing' AND dt.updated_at < $1
ORDER BY dt.updated_at ASC;

-- name: MarkStaleTaskPending :execrows
UPDATE document_tasks SET status = 'pending', retry_count = retry_count + 1, updated_at = now()
WHERE id = $1 AND status = 'processing';

-- name: MarkStaleTaskFailed :execrows
UPDATE document_tasks SET status = 'failed',
    error_message = 'stale task: exceeded max retry attempts', updated_at = now()
WHERE id = $1 AND status = 'processing';
```

`MarkStaleTaskPending` и `MarkStaleTaskFailed` используют `WHERE status = 'processing'` как CAS: если другой worker уже забрал задачу, `rowsAffected = 0` и watchdog молча пропускает.

Также `GetDocumentTask` и `ListTasksByDocument` переведены на `dt.*` вместо явного перечисления колонок — это предотвращает генерацию sqlc отдельных `*Row`-типов при JOIN, что ломало бы service-код.

## `internal/watchdog`

### Дизайн

```go
type Watchdog struct {
    q   repository.Querier
    pub pythonPublisher   // consumer-side interface: только Process()
    cfg *config.Config
    log *slog.Logger
}
```

- `Start(ctx)` — блокирующий цикл на `time.Ticker` с интервалом `cfg.WatchdogInterval`.
- `runOnce(ctx)` — основная логика: выборка задач старше `cfg.WatchdogThreshold`, обработка каждой.
- `processTask()` — если `RetryCount >= MaxRetries` → `MarkStaleTaskFailed`; иначе → `MarkStaleTaskPending` + `Process()`.
- Ошибки публикации в Redis логируются и не пробрасываются (best-effort).

### Конфиг

```
WATCHDOG_INTERVAL    = 2m    # частота проверок
WATCHDOG_THRESHOLD   = 10m   # задача считается зависшей
WATCHDOG_MAX_RETRIES = 5     # после этого — permanent fail
```

---

## Подключение в `cmd/api/main.go`

```go
watchdogCtx, watchdogCancel := context.WithCancel(context.Background())
defer watchdogCancel()
go watchdog.Start(watchdogCtx, srv.DB(), srv.PythonPublisher(), cfg, log)
```

Добавлены геттеры `Server.DB()` и `Server.PythonPublisher()` для передачи зависимостей без нарушения инкапсуляции.

# 2026-04-26 (продолжение) — Артефакты документов

## Watchdog: покрытие статуса pending

Сторожевой таймер перезапускал только зависшие задачи со статусом `processing`. Добавили `pending` (задача не была подхвачена воркером).

- **Миграция 000003** — расширен partial index `idx_document_tasks_stale`: `WHERE status IN ('pending', 'processing')`.
- `ListStaleTasks`, `MarkStaleTaskPending`, `MarkStaleTaskFailed` обновлены аналогично.
- Тест `TestRunOnce_RequeuesStaleTaskInPendingStatus` добавлен → итого 6 кейсов в `watchdog_test.go`.

## Артефакты документов

Воркер порождает производные документы: convert → markdown, anonymize → анонимизированный + карта сущностей. Реализована полная цепочка хранения артефактов.

### Миграция 000004

- `documents.artifact_kind VARCHAR(50) NULL` — тип артефакта (`convert_md`, `anonymize_doc`, `anonymize_entities`).
- FK `documents_parent_id_fkey`: `ON DELETE SET NULL` → `ON DELETE CASCADE`.
- Partial index `idx_documents_root ON documents(organization_id, created_at DESC) WHERE parent_id IS NULL`.
