# 2026-04-29 — День 11: Батч-запрос задач по списку document_id

## Контекст

Фронтенд хотел polling задач сразу для нескольких документов (страница проекта),
чтобы избежать N отдельных запросов к `GET /api/v1/tasks?document_id=...`.

---

## Что сделано

### SQL — `sql/query/document_task.sql`

Добавлен запрос `ListTasksByDocuments`:

```sql
-- name: ListTasksByDocuments :many
SELECT dt.*
FROM document_tasks AS dt
JOIN documents AS d ON d.id = dt.document_id
WHERE dt.document_id = ANY($1::uuid[])
  AND d.organization_id = $2
ORDER BY dt.document_id, dt.created_at DESC;
```

Tenant-isolation реализована через JOIN с `documents` — чужие документы не попадают в ответ.

После добавления запущены `make sqlc` и `make mock` — сгенерированы `internal/repository/document_task.sql.go`, обновлён `querier.go` и `internal/store/mock/mock_querier.go`.

> **Замечание:** sqlc сгенерировал поле `Column1 []uuid.UUID` (не `DocumentIds`) из-за синтаксиса `ANY($1::uuid[])`.

### Сервис — `internal/service/service_document_task.go`

```go
func (s *DocumentTaskService) ListByDocuments(ctx context.Context, documentIDs []uuid.UUID, orgID uuid.UUID) ([]repository.DocumentTask, error)
```

Оборачивает `repo.ListTasksByDocuments`, ошибки пробрасывает как `errs.CodeInternalError`.

### Хендлер — `internal/server/handler_document_task.go`

`ListDocumentTasks` расширен: теперь принимает как `document_id` (одиночный UUID), так и `document_ids` (список через запятую, максимум 100).

Логика:
- Оба параметра одновременно → 400
- Ни одного → 400
- `document_ids` → парсинг, валидация каждого UUID, батч-запрос
- `document_id` → старое поведение (обратная совместимость)

Добавлен импорт `"strings"`.
