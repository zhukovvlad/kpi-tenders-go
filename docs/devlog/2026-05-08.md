### 🟡 Gap 4 — `SiteListItem` с агрегированными полями на `/sites/root` и `/sites/:id/children`

Фронтенд ожидал обогащённый объект вместо `repository.ConstructionSite` как есть.

**SQL-запросы (3 новых):**

```sql
-- GetSiteAncestors: рекурсивный CTE вверх по дереву parent_id
-- Возвращает имена от корня к сайту (ORDER BY depth DESC)
WITH RECURSIVE ancestry AS (
    SELECT cs.id, cs.name, cs.parent_id, 0 AS depth ...
    UNION ALL
    SELECT p.id, p.name, p.parent_id, a.depth + 1 FROM construction_sites p JOIN ancestry a ON p.id = a.parent_id
)
SELECT name FROM ancestry ORDER BY depth DESC;

-- ListSiteExtractedCounts: COUNT документов с extracted_data по batch сайтов
SELECT d.site_id, COUNT(ded.id)::bigint AS extracted_count
FROM documents d JOIN document_extracted_data ded ...
WHERE d.site_id = ANY(sqlc.arg(site_ids)::uuid[])
GROUP BY d.site_id;

-- ListSiteContractKinds: уникальные contract_kinds документов по batch сайтов
SELECT d.site_id, ck.id, ck.display_name, ck.is_active
FROM documents d JOIN document_contract_kinds ck ON ck.id = d.contract_kind_id
WHERE d.site_id = ANY(sqlc.arg(site_ids)::uuid[])
GROUP BY d.site_id, ck.id, ...;
```

**Сервис:**
- Новые DTOs: `SiteContractKindSummary`, `SiteListItem` (embeds `repository.ConstructionSite` + `Breadcrumbs []string`, `ContractKinds []SiteContractKindSummary`, `AggregateStatus string`, `ExtractedCount int64`, `InflationPct *float64`).
- `ListRootWithMeta(ctx, orgID)` — корневые сайты без хлебных крошек.
- `ListChildrenWithMeta(ctx, orgID, parentID)` — дочерние; сначала `GetSiteAncestors(parentID)` для breadcrumbs родителя.
- `enrichSites(...)` — батчевый запрос статусов (`ListSiteStatusesByOrg`), counts, kinds; сборка `[]SiteListItem`.

**Хендлеры:** `ListRootConstructionSites` и `ListConstructionSitesByParent` переключены на новые методы сервиса. Тип ответа изменился с `[]ConstructionSite` на `[]SiteListItem`.

**Нюанс pgtype:** `ListSiteContractKindsRow.SiteID` и `ListSiteExtractedCountsRow.SiteID` — `pgtype.UUID` (sqlc выводит из GROUP BY context). Map-ключи строятся через `.Bytes`: `countMap[c.SiteID.Bytes]`.

### 🟡 Gap 5 — `GET /api/v1/sites/:id/events`

Новый эндпойнт с человекочитаемыми событиями (в отличие от сырого `audit-log`).

**SQL:**
```sql
-- ListSiteEventsBySite: audit_log + JOIN users для actor_name
SELECT sal.*, COALESCE(u.first_name || ' ' || u.last_name, 'Система') AS actor_name
FROM site_audit_log sal LEFT JOIN users u ON sal.actor_user_id = u.id
WHERE sal.site_id = ... ORDER BY sal.created_at DESC, sal.id DESC
LIMIT ... OFFSET ...;
```
`actor_name` — `interface{}` в sqlc (COALESCE нестатически выводим); в сервисе: `fmt.Sprintf("%v", r.ActorName)`.

**Сервис (`service_site_audit.go`):**
- DTO `SiteEvent{ID, SiteID, Kind, ActorName, Message, OccurredAt}`.
- `ListEventsBySite(ctx, siteID, orgID, limit, offset)`.
- `buildEventMessage(eventType, payload)` — switch по типу события, для `document_uploaded` достаёт `document_name` из payload, для `extraction_failed` — `error`.

**Хендлер:** `ListSiteEvents` в `handler_site_audit.go`; проверяет принадлежность сайта оргу через `constructionSiteService.Get` перед запросом событий. Маршрут: `GET /api/v1/sites/:id/events`.

## Ловушки и особенности

- **Рекурсивный CTE, ambiguous column.** Первый `make sqlc` упал с `column reference "id" is ambiguous at construction_site.sql:56`. Причина: в recursive-части CTE оба JOIN-а видны без явного alias. Фикс: переименовать recursive член `FROM construction_sites p` с явным alias `p` и использовать `p.id`, `p.name`, `p.parent_id`.
- **Array param в sqlc.** `sqlc.arg(site_ids)::uuid[]` → Go-тип `[]uuid.UUID`. При пустом срезе сайтов не нужно пропускать запрос — PG вернёт пустой результат.
- **`interface{}` для COALESCE.** sqlc не выводит статический тип из `COALESCE(text_expr, 'literal')`. Обходится через `fmt.Sprintf("%v", r.ActorName)` в сервисе. Альтернатива — `COALESCE(u.first_name, '')::text`, но это не меняет поведение вывода sqlc.

## Фиксы по замечаниям Copilot Code Review (PR #19)

Copilot оставил 4 замечания на PR #19. Все четыре применены.

### 1. `full_name` в `site_audit_log.sql`

Ошибка: `COALESCE(u.first_name || ' ' || u.last_name, 'Система')` — конкатенация двух nullable-полей даёт NULL, если любое из них NULL. Правильно использовать готовое поле `full_name`.

```sql
-- было
COALESCE(u.first_name || ' ' || u.last_name, 'Система') AS actor_name
-- стало
COALESCE(u.full_name, 'Система') AS actor_name
```

### 2–3. Пагинация `GET /api/v1/documents/:id/extraction-requests`

Эндпойнт возвращал все записи без LIMIT. Добавлены `?limit` и `?offset` (по умолчанию 20 и 0, max 100).

- SQL: `LIMIT sqlc.arg(limit_) OFFSET sqlc.arg(offset_)` в `ListExtractionRequestsByDocument`.
- Сервис: `ListRequestsByDocument(ctx, docID, orgID, limit, offset int32)`.
- Хендлер: inline-парсинг `strconv.ParseInt` по аналогии с `handler_site_audit.go`.

### 4. `enrichSites`: замена full-tenant запроса на site-scoped

`ListSiteStatusesByOrg` грузил статусы всех сайтов организации, тогда как нужны статусы только текущего батча. Добавлен новый SQL-запрос:

  AND site_id = ANY(sqlc.arg(site_ids)::uuid[]);

`enrichSites` переключён на `ListSiteStatusesBySiteIds`. `make mock` регенерирован.

### Обновлённые тесты

- `handler_extraction_request_test.go`: мок `ListExtractionRequestsByDocument` дополнен `Limit: 20, Offset: 0`.
- `handler_construction_site_test.go`: мок `ListSiteStatusesByOrg` заменён на `ListSiteStatusesBySiteIds` с явным `SiteIds: []uuid.UUID{...}`.

## Фиксы по замечаниям CodeRabbit (PR #19, второй раунд)

CodeRabbit оставил 6 замечаний. Все шесть применены.

### 1. actor_name: nil bug

`fmt.Sprintf("%v", r.ActorName)` при `interface{}(nil)` возвращал строку `"<nil>"` вместо `"Система"`. После того как `make sqlc` сгенерировал `ActorName` как `string` (COALESCE гарантирует непустое значение), код упрощён до:

```go
actorName := r.ActorName
if actorName == "" {
    actorName = "Система"
}
```

### 2. CTE: утечка тенанта в рекурсии

В `GetSiteAncestors` рекурсивная ветка CTE не фильтровала по `organization_id`, что теоретически позволяло обходить tenant isolation по цепочке parent_id. Добавлена проверка:

```sql
FROM construction_sites p
JOIN ancestry a ON p.id = a.parent_id
WHERE p.organization_id = a.organization_id  -- добавлено
```

### 3. Нестабильный ORDER BY в `ListExtractionRequestsByDocument`

`ORDER BY created_at DESC` нестабилен при совпадающих значениях. Добавлена вторичная сортировка:

```sql
ORDER BY created_at DESC, id DESC
```

### 4. `parseSiteAuditPagination` — общий хелпер

Дублирование inline-парсинга `?limit` и `?offset` в `ListSiteAuditLog` и `ListSiteEvents`. Вынесено в общий хелпер:

```go
func parseSiteAuditPagination(c *gin.Context) (limit, offset int32, appErr error)
```

Оба хендлера переключены на `parseSiteAuditPagination(c)`.

### 5. `mq.AssertExpectations(t)` в тестах сайтов

Два теста в `handler_construction_site_test.go` не вызывали `mq.AssertExpectations(t)` после проверок ответа. Добавлено в оба: `TestListRootConstructionSites_Success_ReturnsSiteListItems` и `TestListConstructionSitesByParent_Success`.

### 6. CLAUDE.md: `/sites/:id/children` документация

Описание эндпойнта дополнено полным списком полей `SiteListItem`.

`make test-unit → ok` по всем пакетам после всех шести фиксов.
