### Что сделано

**1. Миграция: `bin` → `inn`**
- `sql/migrations/000001_init_schema.up.sql` — переименовано поле `bin TEXT NOT NULL UNIQUE` в `inn TEXT UNIQUE`
- Поле стало **nullable** (убрано `NOT NULL`) — ИНН необязателен на этапе регистрации
- Добавлен комментарий: «ИНН организации (10 цифр для юридических лиц)»
- Причина: проект российский, казахстанский БИН не применим

**2. Модель (`internal/repository/models.go`)**
- `Organization.Bin string` → `Organization.Inn pgtype.Text` (nullable)

**3. SQL-запросы (`sql/query/organization.sql`)**
- `CreateOrganization` — `INSERT` с `name`, `inn`
- `GetOrganizationByID` — по первичному ключу
- `GetOrganizationByINN` — для проверки уникальности / поиска
- `UpdateOrganization` — `name`, `inn`, `updated_at`

**4. SQL-запросы (`sql/query/user.sql`)**

**5. Кодогенерация**
- `make sqlc` → сгенерированы `internal/repository/organization.sql.go`, обновлены `user.sql.go` и `querier.go`

**6. OrganizationService (`internal/service/organization.go`)**
- `Register(ctx, RegisterParams)` — транзакционно создаёт организацию + первого пользователя с ролью `admin`, хэширует пароль через bcrypt
- `GetByID(ctx, id)` — возвращает `ErrOrgNotFound` при `pgx.ErrNoRows`
- `Update(ctx, id, name, inn)` — обновляет поля, распознаёт конфликт ИНН
- `Delete(ctx, id)` — удаление с каскадом
- Сентинельные ошибки: `ErrOrgNotFound`, `ErrEmailTaken`, `ErrINNTaken`
- Вспомогательная `isUniqueViolation` — детектирует PostgreSQL SQLSTATE `23505` по имени constraint

**7. Organization Handlers (`internal/server/handler_organization.go`)**
- `POST /api/v1/auth/register` — публичный; создаёт орг + admin-пользователя, сразу выдаёт auth-куки
- `GET /api/v1/organizations/:id` — только своя организация (сверка `orgID` из токена)
- `PATCH /api/v1/organizations/:id` — только `admin`, только своя орг
- `DELETE /api/v1/organizations/:id` — только `admin`, каскадное удаление, сбрасывает куки
- Вспомогательная `orgIDFromContext` — извлекает `orgID` из Gin-контекста

**8. Server (`internal/server/server.go`)**
- `organizationService *service.OrganizationService` добавлен в структуру `Server`
- `NewOrganizationService` вызывается в `NewServer`
- Зарегистрированы маршруты:
  ```text
  POST   /api/v1/auth/register
  GET    /api/v1/organizations/:id
  PATCH  /api/v1/organizations/:id
  DELETE /api/v1/organizations/:id
  ```

### Маршруты API

| Метод | Путь | Auth | Роль |
|-------|------|------|------|
| `POST` | `/api/v1/auth/register` | — | — |
| `GET` | `/api/v1/organizations/:id` | cookie | любая |
| `PATCH` | `/api/v1/organizations/:id` | cookie | admin |
| `DELETE` | `/api/v1/organizations/:id` | cookie | admin |

### Конвенции именования файлов в `internal/server/`

Хэндлеры переименованы по схеме `handler_<domain>.go`:

| Старое имя | Новое имя |
|---|---|
| `auth.go` | `handler_auth.go` |
| `handlers.go` | `handler_document.go` |
| `organization.go` | `handler_organization.go` |

`server.go`, `middleware.go`, `health.go` — без изменений. Всё остаётся в пакете `server`, хэндлеры — методы на `*Server`.

### Принятые решения

- **Транзакция при регистрации** — орг и первый пользователь создаются атомарно; при ошибке на любом шаге откатывается всё
- **`inn` nullable** — компания может зарегистрироваться без ИНН и добавить его позже через `PATCH`
- **`admin` при регистрации** — первый пользователь всегда получает роль `admin`; приглашать остальных — задача следующего этапа
- **Проверка `id == orgID` из токена** — пользователь не может получить или изменить чужую организацию без отдельного endpoint для superadmin
- **`DELETE` сбрасывает куки** — после удаления организации продолжать сессию бессмысленно

## 2026-04-16 — Рефакторинг: централизованная обработка ошибок (`pkg/errs`)

### Мотивация

Обработка ошибок в сервисах была тесно связана с деталями БД (`pgconn`, `pgx`), а хэндлеры содержали громоздкие `switch/errors.Is` блоки, дублируя маппинг ошибок → HTTP-статусы. Нарушался принцип инкапсуляции: сервис знал об HTTP-кодах косвенно, хэндлер — о SQL деталях напрямую.

### Что сделано

**1. Пакет `pkg/errs` (`pkg/errs/errs.go`)**
- Тип `Code string` — машинно-читаемый идентификатор для фронтенда
- Стандартные коды: `internal_error`, `not_found`, `conflict`, `validation_failed`, `unauthorized`, `forbidden`
- Структура `Error{Code, Message, Err}`:
  - `Err` помечен `json:"-"` — не попадает в JSON-ответ, только для логирования
  - Реализует интерфейс `error` и `Unwrap()` для совместимости с `errors.As`/`errors.Is`
- Конструктор `New(code, msg, err)`

**2. `internal/server/errors.go` — маппинг ошибок в HTTP**
- Метод `(s *Server) respondWithError(c *gin.Context, err error)` — единственное место трансляции ошибок в HTTP-ответ
- Использует `errors.As` для извлечения `*errs.Error`
- Маппинг `errs.Code` → HTTP-статус через таблицу `codeToStatus`
- Логирование через `slog`: бизнес-ошибки на уровне `Debug`, `internal_error` с полным текстом причины на уровне `Error`
- Fallback для неизвестных типов ошибок: 500 + `Error`-лог без раскрытия деталей клиенту

**3. `internal/pgutil/pgutil.go` — изоляция pgconn**
- Хелпер `IsUniqueViolation(err, constraint)` выделен в отдельный пакет `internal/pgutil`
- Это единственное место в кодовой базе с импортом `pgconn` — нарушение инкапсуляции локализовано
- (Промежуточное имя `internal/service/db_utils.go` использовалось до следующего шага — переименование в `internal/pgutil` описано ниже)

**4. `internal/service/service_organization.go` — рефакторинг**
- Удалены sentinel-ошибки: `ErrOrgNotFound`, `ErrEmailTaken`, `ErrINNTaken`, `ErrInvalidINN`
- Удалены импорты `fmt` и `pgconn`
- Все `fmt.Errorf(...)` заменены на `errs.New(errs.CodeXxx, "...", err)`
- Валидация ИНН возвращает `errs.New(errs.CodeValidationFailed, ...)`

**5. `internal/server/handler_organization.go` — упрощение хэндлеров**
- Удалён импорт `"errors"` и все `errors.Is(err, service.ErrXxx)` switch-блоки
- Каждый блок обработки ошибок заменён одним вызовом `s.respondWithError(c, err)`
- Хэндлеры больше не знают об SQL и не содержат логику маппинга ошибок

### Принятые решения

- **`errors.As` вместо `errors.Is`** — позволяет работать с обёрнутыми `*errs.Error` без потери совместимости с цепочками ошибок
- **Оригинальная ошибка в `Err`, а не в `Message`** — клиент получает только безопасное сообщение; полный стек доступен только в логах
- **`slog` уровни по коду ошибки** — бизнес-ошибки (404, 409, 400) не засоряют `Error`-лог; инфраструктурные проблемы сразу видны
- **Fallback `respondWithError`** — защита от случайно прокинутых «сырых» ошибок без обёртки `errs.New`

## 2026-04-16 — Рефакторинг: переименование сервисов и выделение `internal/pgutil`

### Что сделано

**1. Переименование файлов в `internal/service/`**

| Было | Стало |
|---|---|
| `auth.go` | `service_auth.go` |
| `document.go` | `service_document.go` |
| `organization.go` | `service_organization.go` |

`db_utils.go` не переименовывался — он не является файлом сервисной логики (см. ниже).

**2. Новый пакет `internal/pgutil`**
- `db_utils.go` удалён из `internal/service`

### Принятые решения

- **`db_utils.go` → `internal/pgutil`** — функция детектирования PostgreSQL-ошибок не является бизнес-логикой; её место в инфраструктурном пакете, изолирующем зависимость от `pgconn`
- **Префикс `service_`** — явное соответствие между именем файла и его содержимым; удобно при навигации в IDE без дерева файлов

## 2026-04-16 — Code review: устранение замечаний (PR #3)

### Замечания и решения

**1. Противоречие в девлоге про `db_utils.go`** *(Copilot + CodeRabbit)*
Оба предыдущих раздела описывали файл в разных состояниях. Исправлено: пункт #3 первого раздела обновлён — теперь сразу ссылается на финальное место `internal/pgutil`, с пометкой о промежуточном шаге.

**2. Непоследовательные форматы ошибок в хэндлерах** *(Copilot)*
Часть ответов шла через `respondWithError` (`{error:{code,message}}`), часть — через `gin.H{"error":"..."}`. Исправлено во всех файлах сервера:
- `handler_organization.go` — все `c.JSON` + `gin.H` заменены на `respondWithError(c, errs.New(...))`, добавлен `CodeForbidden`
- `handler_auth.go` — binding-ошибки, все ветки `errors.Is` и прямые `c.JSON` заменены на `respondWithError`
- `handler_document.go` — `GetDocument` стандартизирован
- `middleware.go` — `AbortWithStatusJSON` использует `errorResponse{}` envelope вместо `gin.H`
- `orgIDFromContext` — использует `errorResponse{}` envelope напрямую
- `service_auth.go` — `ErrInvalidCredentials`, `ErrInvalidToken` и `fmt.Errorf` заменены на `errs.New`; добавлен `CodeForbidden` в `pkg/errs`

**3. Утечка нестандартного кода при неизвестном `errs.Code`** *(Copilot)*
Если `Code` не присутствует в `codeToStatus`, ответ клиенту теперь всегда использует `CodeInternalError` + 500, а оригинальный код логируется на уровне `Error`.

**4. PII в логах для не-internal ошибок** *(Copilot)*
Для ошибок с кодом, отличным от `internal_error`, причина (`Err`) больше не логируется. PostgreSQL-детали уникального нарушения (содержат значение поля) теперь не попадают в лог.
