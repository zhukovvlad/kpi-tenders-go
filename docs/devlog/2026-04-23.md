# Дневник разработки — go-kpi-tenders

## 2026-04-23 — День 7: S3-загрузка файлов, пакет `internal/storage`

### Контекст

Python-воркер уже умеет читать и писать файлы в MinIO (см. `app/storage/minio_client.py`).  
Теперь нужно, чтобы Go-бэкенд сам принимал файл от клиента, загружал его в S3 и создавал запись документа — без посредников.

---

### Что сделано

**1. Пакет `internal/storage`**

Создан новый пакет-обёртка над `minio-go/v7`:

```go
// Client wraps MinIO and exposes a minimal interface for document storage.
type Client struct {
    mc     *minio.Client
    bucket string
}
```

Публичные методы:

| Метод | Описание |
|-------|----------|
| `New(cfg)` | Инициализирует клиент из `config.Config` (S3_*) |
| `Upload(ctx, r, size, filename, contentType)` | Загружает файл, возвращает `storagePath` |
| `PresignedURL(ctx, storagePath, ttl)` | Генерирует временный GET URL |

**Формат `storagePath`:** `"{bucket}/{uuid}.{ext}"` — идентичен Python-стороне (`parse_storage_path` в `minio_client.py`). Таким образом, файл, загруженный через Go, может быть сразу скачан Python-воркером без дополнительных преобразований.

`PresignedGetObject` в minio-go генерирует подписанный URL локально (HMAC, без сетевого вызова) — это важно для unit-тестирования.

**2. Зависимость**

```bash
go get github.com/minio/minio-go/v7@latest
```

**3. Интеграция `storageClient` в `Server`**

В `Server` добавлено поле:

```go
storageClient storageClient // интерфейс (server.storageClient), описан в storage_iface.go
```

Инициализация в `NewServer()` — ленивая и non-fatal: если `S3AccessKey`/`S3SecretKey` не заданы (тестовое окружение), клиент остаётся `nil`, сервер стартует. Upload-эндпоинт в этом случае деградирует с 500.

**4. Хендлер `UploadDocument`**

Добавлен в `internal/server/handler_document.go`:

- Принимает `multipart/form-data`, лимит файла **100 MiB** (`maxFileSize`); лимит всего request body **101 MiB** (`maxRequestBodySize`, с запасом на multipart-оверхед)
- Обязательное поле: `file`
- Опциональные: `site_id`, `parent_id` (UUID)
- Последовательность: проверка storageClient → аутентификация → парсинг файла → валидация `site_id`/`parent_id` → Upload → `documentService.Create` → `201 Created`
- MIME-тип берётся из заголовка `Content-Type` части; fallback: `application/octet-stream`

**5. Роутинг**

```go
documents.POST("/upload", s.UploadDocument)
```

Маршрут добавлен рядом с существующим `POST /api/v1/documents` внутри `protected` группы (требует JWT).

### День 7 (продолжение): безопасные имена объектов S3 — `SafeExt`

**Проблема:** S3 object key строится как `uuid + ext`. Если расширение содержит кириллицу (`.пдф`) или другие non-ASCII символы — это может сломать совместимость с некоторыми S3-провайдерами.

**Решение:** новая экспортируемая функция `SafeExt` в `internal/storage/client.go`:

```go
func SafeExt(name string) string {
    ext := strings.ToLower(filepath.Ext(name))
    if len(ext) <= 1 || len(ext) > 10 {
        return ""
    }
    // Allow only [a-z0-9] after the leading dot.
    for _, r := range ext[1:] {
        if (r < 'a' || r > 'z') && (r < '0' || r > '9') {
            return ""
        }
    }
    return ext
}
```

Правила отброса:
- голая точка (`"."`) — `len <= 1`
- расширение длиннее 10 байт
- любой символ после точки, не входящий в `[a-z0-9]` (кириллица, CJK, пробелы, спецсимволы и т.п.)

Примечание: uppercase-буквы не отбрасываются, они нормализуются через `strings.ToLower` перед проверкой (`.PDF` → `.pdf`).

Примеры:
| Имя файла | S3 key |
|-----------|--------|
| `report.pdf` | `uuid.pdf` |
| `report.PDF` | `uuid.pdf` |
| `договор.pdf` | `uuid.pdf` |
| `договор.пдф` | `uuid` (без расширения) |
| `C:\path\file.PDF` | `uuid.pdf` |

`Upload` теперь вызывает `SafeExt` вместо прямого `filepath.Ext`.

**Тесты** — 10 table-driven кейсов в `TestSafeExt` в `client_test.go`:
- ascii lowercase/uppercase, multi-dot, кириллическое расширение, кирилличное имя с ascii расширением, нет расширения, голая точка, слишком длинное расширение, Windows-путь, смешанное расширение.
